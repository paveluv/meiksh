# Test Suite for Intrinsic Utility: read

This test suite covers the **read** intrinsic utility
as specified by POSIX.1-2024 (Section 1.7).

## Table of contents

- [utility: read](#utility-read)

## utility: read

#### NAME

> read — read from standard input into shell variables

#### SYNOPSIS

> `read [-r] [-d delim] var...`

#### DESCRIPTION

> The *read* utility shall read a single logical line from standard input into one or more shell variables.
>
> If the **-r** option is not specified, `<backslash>` shall act as an escape character. An unescaped `<backslash>` shall preserve the literal value of a following `<backslash>` and shall prevent a following byte (if any) from being used to split fields, with the exception of either `<newline>` or the logical line delimiter specified with the **-d** *delim* option (if it is used and *delim* is not `<newline>`); it is unspecified which. If this excepted character follows the `<backslash>`, the *read* utility shall interpret this as line continuation. The `<backslash>` and the excepted character shall be removed before splitting the input into fields. All other unescaped `<backslash>` characters shall be removed after splitting the input into fields.
>
> If standard input is a terminal device and the invoking shell is interactive, *read* shall prompt for a continuation line when it reads an input line ending with a `<backslash>` `<newline>`, unless the **-r** option is specified.
>
> The terminating logical line delimiter (if any) shall be removed from the input. Then, if the shell variable *IFS* (see [*2.5.3 Shell Variables*](docs/posix/md/utilities/V3_chap02.md#253-shell-variables)) is set, and its value is an empty string, the resulting data shall be assigned to the variable named by the first *var* operand, and the variables named by other *var* operands (if any) shall be set to the empty string. No other processing shall be performed in this case.
>
> If *IFS* is unset, or is set to any non-empty value, then a modified version of the field splitting algorithm specified in [*2.6.5 Field Splitting*](docs/posix/md/utilities/V3_chap02.md#265-field-splitting) shall be applied, with the modifications as follows:
>
> 1. The input to the algorithm shall be the logical line (minus terminating delimiter) that was read from standard input, and shall be considered as a single initial field, all of which resulted from expansions, with any escaped byte and the preceding `<backslash>` escape character treated as if they were the result of a quoted expansion, and all other bytes treated as if they were the results of unquoted expansions.
> 2. The loop over the contents of that initial field shall cease when either the input is empty or *n* output fields have been generated, where *n* is one less than the number of *var* operands passed to the *read* utility. Any remaining input in the original field being processed shall be returned to the *read* utility "unsplit"; that is, unmodified except that any leading or trailing *IFS* white space, as defined in [*2.6.5 Field Splitting*](docs/posix/md/utilities/V3_chap02.md#265-field-splitting), shall be removed.
>
> The specified *var* operands shall be processed in the order they appear on the command line, and the output fields generated by the field splitting algorithm shall be used in the order they were generated, by repeating the following checks until neither is true:
>
> - If more than one *var* operand is yet to be processed and one or more output fields are yet to be used, the variable named by the first unprocessed *var* operand shall be assigned the value of the first unused output field.
> - If exactly one *var* operand is yet to be processed and there was some remaining unsplit input returned from the modified field splitting algorithm, the variable named by the unprocessed *var* operand shall be assigned the unsplit input.
>
> If there are still one or more unprocessed *var* operands, each of the variables names by those operands shall be assigned an empty string.
>
> Note that in the case where just one *var* operand is given on the *read* command line, the modified field splitting algorithm ceases after producing zero output fields and simply returns the original input field, with any leading and trailing *IFS* white space removed, as unsplit input. This unsplit input is assigned to the variable named by the *var* operand.
>
> The setting of variables specified by the *var* operands shall affect the current shell execution environment; see [*2.13 Shell Execution Environment*](docs/posix/md/utilities/V3_chap02.md#213-shell-execution-environment). An error in setting any variable (such as if a *var* has previously been marked *readonly*) shall be considered an error of *read* processing, and shall result in a return value greater than one. Variables named before the one generating the error shall be set as described above; it is unspecified whether variables named later shall be set as above, or *read* simply ceases processing when the error occurs, leaving later named variables unaltered. If *read* is called in a subshell or separate utility execution environment, such as one of the following:
>
> ```
> (read foo)
> nohup read ...
> find . -exec read ... \;
> ```
>
> it shall not affect the shell variables in the caller's environment.
>
> If end-of-file is detected before a terminating logical line delimiter is encountered, the variables specified by the *var* operands shall be set as described above and the exit status shall be 1.

#### OPTIONS

> The *read* utility shall conform to XBD [*12.2 Utility Syntax Guidelines*](docs/posix/md/basedefs/V1_chap12.md#122-utility-syntax-guidelines).
>
> The following options shall be supported:
>
> - **-d** *delim*: If *delim* consists of one single-byte character, that byte shall be used as the logical line delimiter. If *delim* is the null string, the logical line delimiter shall be the null byte. Otherwise, the behavior is unspecified.
> - **-r**: Do not treat a `<backslash>` character in any special way. Consider each `<backslash>` to be part of the input line.

#### OPERANDS

> The following operand shall be supported:
>
> - *var*: The name of an existing or nonexisting shell variable. If a *var* operand names the variable *IFS ,* the behavior is unspecified. If a *var* operand names one of the variables *LANG ,* *LC_CTYPE ,* or *LC_ALL* and the new value assigned to the variable would change how the bytes in *IFS* form characters, or which characters in *IFS* are considered to be *IFS* white space (see [*2.6.5 Field Splitting*](docs/posix/md/utilities/V3_chap02.md#265-field-splitting)), it is unspecified what effects, if any, the change has on how *read* performs field splitting.

#### STDIN

> If the **-d** *delim* option is not specified, or if it is specified and *delim* is not the null string, the standard input shall contain zero or more bytes (which need not form valid characters) and shall not contain any null bytes.
>
> If the **-d** *delim* option is specified and *delim* is the null string, the standard input shall contain zero or more bytes (which need not form valid characters).

#### INPUT FILES

> None.

#### ENVIRONMENT VARIABLES

> The following environment variables shall affect the execution of *read*:
>
> - *IFS*: Determine the internal field separators used to delimit fields; see [*2.5.3 Shell Variables*](docs/posix/md/utilities/V3_chap02.md#253-shell-variables).
> - *LANG*: Provide a default value for the internationalization variables that are unset or null. (See XBD [*8.2 Internationalization Variables*](docs/posix/md/basedefs/V1_chap08.md#82-internationalization-variables) for the precedence of internationalization variables used to determine the values of locale categories.)
> - *LC_ALL*: If set to a non-empty string value, override the values of all the other internationalization variables.
> - *LC_CTYPE*: Determine the locale for the interpretation of sequences of bytes of text data as characters (for example, single-byte as opposed to multi-byte characters in arguments).
> - *LC_MESSAGES*: Determine the locale that should be used to affect the format and contents of diagnostic messages written to standard error.
> - *NLSPATH*: Determine the location of messages objects and message catalogs.
> - *PS2*: Provide the prompt string that an interactive shell shall write to standard error when a line ending with a `<backslash>` `<newline>` is read and the **-r** option was not specified.

#### ASYNCHRONOUS EVENTS

> Default.

#### STDOUT

> Not used.

#### STDERR

> The standard error shall be used for diagnostic messages and prompts for continued input.

#### OUTPUT FILES

> None.

#### EXTENDED DESCRIPTION

> None.

#### EXIT STATUS

> The following exit values shall be returned:
>
> - 0: Successful completion.
> - 1: End-of-file was detected.
> - \>1: An error occurred.

#### CONSEQUENCES OF ERRORS

> Default.

*The following sections are informative.*

#### APPLICATION USAGE

> This utility is required to be intrinsic. See [*1.7 Intrinsic Utilities*](docs/posix/md/utilities/V3_chap01.md#17-intrinsic-utilities) for details.
>
> The **-r** option is included to enable *read* to subsume the purpose of the *line* utility, which is not included in POSIX.1-2024.
>
> The **-d** *delim* option enables reading up to an arbitrary single-byte delimiter. When *delim* is the null string, the delimiter is the null byte and this allows *read* to be used to process null-terminated lists of pathnames (as produced by the [*find*](docs/posix/md/utilities/find.md) **-print0** primary), with correct handling of pathnames that contain `<newline>` characters. Note that in order to specify the null string as the delimiter, **-d** and *delim* need to be specified as two separate arguments. Implementations differ in their handling of `<backslash>` for line continuation when **-d** *delim* is specified (and *delim* is not `<newline>`); some treat `<backslash>`*delim* (or `<backslash>``<NUL>` if *delim* is the null string) as a line continuation, whereas others still treat `<backslash>``<newline>` as a line continuation. Consequently, portable applications need to specify **-r** whenever they specify **-d** *delim* (and *delim* is not `<newline>`).
>
> When reading a pathname it is inadvisable to use the contents of the first *var* operand, if non-empty, when the exit status of *read* is 1, as it is likely the result of the command used to generate the list of pathnames (for example [*find*](docs/posix/md/utilities/find.md) with **-print** or **-print0**) being terminated after it has written a partial pathname, and consequently using it could result in the wrong pathname being processed.
>
> Since the *var* operands are processed in the order specified on the command line, if any variable name is specified more than once as a *var* operand, the last assignment made is the one that is in effect when *read* returns, including when an empty string is assigned because no field data was available.

#### EXAMPLES

> The following command:
>
> ```
> while read -r xx yy
> do
>     printf "%s %s\n" "$yy" "$xx"
> done < input_file
> ```
>
> prints a file with the first field of each line moved to the end of the line.

#### RATIONALE

> The *read* utility historically has been a shell built-in. It was separated off into its own utility to take advantage of the richer description of functionality introduced by this volume of POSIX.1-2024.
>
> Since *read* affects the current shell execution environment, it is required to be intrinsic. If it is called in a subshell or separate utility execution environment, such as one of the following:
>
> ```
> (read foo)
> nohup read ...
> find . -exec read ... \;
> ```
>
> it does not affect the shell variables in the environment of the caller.
>
> Earlier versions of this standard required the standard input to be a text file, and therefore the results were undefined if the input was not empty and end-of-file was detected before a `<newline>` character was encountered. However, all of the most popular shell implementations have been found to have consistent behavior in this case, and so the behavior is now specified and the requirement for standard input to be a text file has been relaxed to allow non-empty input that does not end with a `<newline>`.

### Tests

#### Test: basic single word

`read` assigns a single input word to the variable.

```
begin test "basic single word"
  script
    echo "hello" > tmp_read_input
    read var < tmp_read_input
    echo "$var"
  expect
    stdout "hello"
    stderr ""
    exit_code 0
end test "basic single word"
```

#### Test: two variables two fields

Input is split into fields; each variable gets one field.

```
begin test "two variables two fields"
  script
    echo "one two" > tmp_read_input
    read a b < tmp_read_input
    echo "$a $b"
  expect
    stdout "one two"
    stderr ""
    exit_code 0
end test "two variables two fields"
```

#### Test: fewer vars than fields last gets remainder

When there are more fields than variables, the last variable
receives the remaining fields.

```
begin test "fewer vars than fields last gets remainder"
  script
    echo "a b c d" > tmp_read_input
    read x y < tmp_read_input
    echo "$x|$y"
  expect
    stdout "a[|]b c d"
    stderr ""
    exit_code 0
end test "fewer vars than fields last gets remainder"
```

#### Test: read -r treats backslashes literally

With `-r`, backslash characters are treated as literal.

```
begin test "read -r treats backslashes literally"
  script
    printf 'hello\\world\n' > tmp_read_input
    read -r var < tmp_read_input
    printf '%s' "$var"
  expect
    stdout "hello\\world"
    stderr ""
    exit_code 0
end test "read -r treats backslashes literally"
```

#### Test: read at EOF returns non-zero

When `read` encounters end-of-file, it returns a non-zero exit status.

```
begin test "read at EOF returns non-zero"
  script
    printf "" | read var
    echo $?
  expect
    stdout "[1-9].*"
    stderr ""
    exit_code 0
end test "read at EOF returns non-zero"
```

#### Test: basic multi-word

When a single variable is given, `read` assigns the entire input line (including spaces) to that variable.

```
begin test "basic multi-word"
  script
    echo 'hello world' | { read line; echo "$line"; }
  expect
    stdout "hello world"
    stderr ""
    exit_code 0
end test "basic multi-word"
```

#### Test: only first line consumed by single read

A single `read` invocation consumes only the first line of input; subsequent lines remain available for later reads.

```
begin test "only first line consumed by single read"
  script
    printf 'first
    second
    ' | { read line; echo "$line"; }
  expect
    stdout "first"
    stderr ""
    exit_code 0
end test "only first line consumed by single read"
```

#### Test: backslash-space preserves literal space

Without `-r`, a backslash before a space prevents that space from acting as a field delimiter, so the literal space is preserved in the variable's value.

```
begin test "backslash-space preserves literal space"
  script
    printf 'hello\ world
    ' | { read var; echo "$var"; }
  expect
    stdout "hello world"
    stderr ""
    exit_code 0
end test "backslash-space preserves literal space"
```

#### Test: backslash-backslash produces single backslash

Without `-r`, an escaped backslash (`\\`) in the input collapses to a single literal backslash in the resulting variable.

```
begin test "backslash-backslash produces single backslash"
  script
    printf '%s
    ' 'a\\b' | { read var; printf '%s' "$var"; }
  expect
    stdout "a\\b"
    stderr ""
    exit_code 0
end test "backslash-backslash produces single backslash"
```

#### Test: backslash before regular char removes backslash

Without `-r`, a backslash preceding an ordinary character is removed after field splitting, so only the character itself remains in the output.

```
begin test "backslash before regular char removes backslash"
  script
    printf '%s
    ' 'a\b' | { read var; printf '%s' "$var"; }
  expect
    stdout "ab"
    stderr ""
    exit_code 0
end test "backslash before regular char removes backslash"
```

#### Test: backslash-newline joins lines (line continuation)

Without `-r`, a backslash immediately before a newline acts as a line continuation, causing `read` to join the current line with the next and remove both the backslash and the newline.

```
begin test "backslash-newline joins lines (line continuation)"
  script
    printf 'hello\\
    world
    ' | { read var; echo "$var"; }
  expect
    stdout "helloworld"
    stderr ""
    exit_code 0
end test "backslash-newline joins lines (line continuation)"
```

#### Test: read -r backslash-newline does not join lines

With `-r`, backslash has no special meaning, so a trailing backslash before a newline is kept literally and no line continuation occurs.

```
begin test "read -r backslash-newline does not join lines"
  script
    printf 'hello\\
    world
    ' | { read -r var; echo "$var"; }
  expect
    stdout "hello\\"
    stderr ""
    exit_code 0
end test "read -r backslash-newline does not join lines"
```

#### Test: read -r preserves double backslash

With `-r`, backslashes are not treated as escape characters, so a double backslash in the input is preserved verbatim in the variable.

```
begin test "read -r preserves double backslash"
  script
    printf '%s
    ' 'a\\b' | { read -r var; printf '%s' "$var"; }
  expect
    stdout "a\\\\b"
    stderr ""
    exit_code 0
end test "read -r preserves double backslash"
```

#### Test: multiple spaces treated as single delimiter

Consecutive IFS-whitespace characters between fields are collapsed into a single delimiter during field splitting, so multiple spaces between words still produce exactly two fields.

```
begin test "multiple spaces treated as single delimiter"
  script
    echo 'hello    world' | { read a b; echo "$a $b"; }
  expect
    stdout "hello world"
    stderr ""
    exit_code 0
end test "multiple spaces treated as single delimiter"
```

#### Test: leading whitespace stripped

Leading IFS whitespace in the input is removed before field assignment, so variables do not contain leading spaces or tabs.

```
begin test "leading whitespace stripped"
  script
    echo '   hello world' | { read a b; echo "$a $b"; }
  expect
    stdout "hello world"
    stderr ""
    exit_code 0
end test "leading whitespace stripped"
```

#### Test: trailing whitespace stripped from last field

Trailing IFS whitespace is stripped from the last variable's value, even when there are multiple trailing spaces after the final word.

```
begin test "trailing whitespace stripped from last field"
  script
    echo 'hello world   ' | { read a b; echo "$a $b"; }
  expect
    stdout "hello world"
    stderr ""
    exit_code 0
end test "trailing whitespace stripped from last field"
```

#### Test: fewer vars than fields: last gets remainder

When there are more input fields than variables, the last variable receives the unsplit remainder of the line, including any internal delimiters.

```
begin test "fewer vars than fields: last gets remainder"
  script
    echo 'a b c d' | { read first rest; echo "$first $rest"; }
  expect
    stdout "a b c d"
    stderr ""
    exit_code 0
end test "fewer vars than fields: last gets remainder"
```

#### Test: remainder preserves internal spacing

The unsplit remainder assigned to the last variable preserves the original internal whitespace (including runs of multiple spaces) between fields.

```
begin test "remainder preserves internal spacing"
  script
    echo 'a b   c   d' | { read first rest; echo "$first $rest"; }
  expect
    stdout "a b   c   d"
    stderr ""
    exit_code 0
end test "remainder preserves internal spacing"
```

#### Test: more vars than fields: extras get empty

When there are more variables than input fields, excess variables are set to the empty string.

```
begin test "more vars than fields: extras get empty"
  script
    echo 'hello' | { read a b c; echo "$a:$b:$c"; }
  expect
    stdout "hello::"
    stderr ""
    exit_code 0
end test "more vars than fields: extras get empty"
```

#### Test: more vars than fields: two words three vars

With two input words and three variables, the first two variables get one word each and the third is set to the empty string.

```
begin test "more vars than fields: two words three vars"
  script
    echo 'hello world' | { read a b c; echo "$a:$b:$c"; }
  expect
    stdout "hello:world:"
    stderr ""
    exit_code 0
end test "more vars than fields: two words three vars"
```

#### Test: colon as IFS

Setting IFS to a colon causes `read` to split the input on colons instead of the default whitespace, assigning each colon-delimited segment to a variable.

```
begin test "colon as IFS"
  script
    echo 'a:b:c' | { IFS=: read a b c; echo "$a:$b:$c"; }
  expect
    stdout "a:b:c"
    stderr ""
    exit_code 0
end test "colon as IFS"
```

#### Test: comma as IFS

Setting IFS to a comma causes `read` to split the input on commas, so each comma-separated value is assigned to a separate variable.

```
begin test "comma as IFS"
  script
    echo 'x,y' | { IFS=, read a b; echo "$a $b"; }
  expect
    stdout "x y"
    stderr ""
    exit_code 0
end test "comma as IFS"
```

#### Test: multi-character IFS: colon and comma

When IFS contains multiple non-whitespace characters (here colon and comma), each one independently acts as a field delimiter, so the input is split on both.

```
begin test "multi-character IFS: colon and comma"
  script
    echo 'a:b,c' | { IFS=:, read a b c; echo "$a $b $c"; }
  expect
    stdout "a b c"
    stderr ""
    exit_code 0
end test "multi-character IFS: colon and comma"
```

#### Test: non-whitespace IFS preserves leading/trailing delimiters as empty fields

Non-whitespace IFS characters produce empty fields at the boundaries: a leading or trailing delimiter results in an empty string for the corresponding variable.

```
begin test "non-whitespace IFS preserves leading/trailing delimiters as empty fields"
  script
    echo ':b:' | { IFS=: read a b c; echo "$a:$b:$c"; }
  expect
    stdout ":b:"
    stderr ""
    exit_code 0
end test "non-whitespace IFS preserves leading/trailing delimiters as empty fields"
```

#### Test: empty IFS: entire line goes into first variable

When IFS is set to the empty string, no field splitting occurs, so the entire input line is assigned to the first variable and any remaining variables are set to empty.

```
begin test "empty IFS: entire line goes into first variable"
  script
    echo 'hello world' | { IFS= read a b; echo "$a:$b"; }
  expect
    stdout "hello world:"
    stderr ""
    exit_code 0
end test "empty IFS: entire line goes into first variable"
```

#### Test: empty IFS: no field splitting

With an empty IFS, delimiters like colons that would normally split if present in IFS have no effect; the entire input is treated as a single unsplit field.

```
begin test "empty IFS: no field splitting"
  script
    echo 'a:b:c' | { IFS= read x y; echo "$x:$y"; }
  expect
    stdout "a:b:c:"
    stderr ""
    exit_code 0
end test "empty IFS: no field splitting"
```

#### Test: empty IFS with single variable: entire line assigned

With an empty IFS and a single variable, the full input line including any leading and trailing whitespace is assigned without any stripping or splitting.

```
begin test "empty IFS with single variable: entire line assigned"
  script
    printf '  hello  world
    ' | { IFS= read x; echo "$x"; }
  expect
    stdout "  hello  world.*"
    stderr ""
    exit_code 0
end test "empty IFS with single variable: entire line assigned"
```

#### Test: reading into unset variable creates it

`read` creates the named variable if it does not already exist, setting it to the value read from input.

```
begin test "reading into unset variable creates it"
  script
    echo 'created' | { unset newvar; read newvar; echo "$newvar"; }
  expect
    stdout "created"
    stderr ""
    exit_code 0
end test "reading into unset variable creates it"
```

#### Test: reading into existing variable overwrites it

`read` overwrites any existing value of the named variable with the newly read input.

```
begin test "reading into existing variable overwrites it"
  script
    echo 'new' | { existing=old; read existing; echo "$existing"; }
  expect
    stdout "new"
    stderr ""
    exit_code 0
end test "reading into existing variable overwrites it"
```

#### Test: variable set by read persists

Variables assigned by `read` remain set in the current execution environment and are accessible after the `read` call completes.

```
begin test "variable set by read persists"
  script
    echo 'hello world' | { read a b; echo "$a:$b"; }
  expect
    stdout "hello:world"
    stderr ""
    exit_code 0
end test "variable set by read persists"
```

#### Test: read at EOF clears variable to empty

When `read` reaches end-of-file without encountering a line delimiter, the target variable is set to empty, even if it previously held a value.

```
begin test "read at EOF clears variable to empty"
  script
    val=init
    while read val; do :; done <<EOF
    first
    last
    EOF
    echo "done:$val"
  expect
    stdout "done:"
    stderr ""
    exit_code 0
end test "read at EOF clears variable to empty"
```

#### Test: read in explicit subshell does not affect parent

When `read` is executed inside an explicit subshell `(...)`, the variable assignment does not propagate to the parent shell environment.

```
begin test "read in explicit subshell does not affect parent"
  script
    x=parent
    (echo child | read x)
    echo "$x"
  expect
    stdout "parent"
    stderr ""
    exit_code 0
end test "read in explicit subshell does not affect parent"
```

#### Test: read in pipeline subshell does not affect parent

When `read` runs on the right side of a pipeline, it executes in a subshell, so the variable it sets does not affect the parent shell's environment.

```
begin test "read in pipeline subshell does not affect parent"
  script
    x=parent
    echo child | read x
    echo "$x"
  expect
    stdout "parent"
    stderr ""
    exit_code 0
end test "read in pipeline subshell does not affect parent"
```

#### Test: read into readonly variable returns non-zero

Attempting to `read` into a variable that has been marked `readonly` is
an error and must return a non-zero exit status with a diagnostic on
stderr.

```
begin test "read into readonly variable returns non-zero"
  script
    echo 'val' | { readonly rovar=locked; read rovar; }
  expect
    stdout ""
    stderr ".+"
