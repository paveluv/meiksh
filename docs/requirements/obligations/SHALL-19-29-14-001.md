## Extracted Text

> If the trap name [XSI] or number is invalid, a non-zero exit status shall be returned; otherwise, zero shall be returned. For both interactive and non-interactive shells, invalid signal names [XSI] or numbers shall not be considered an error and shall not cause the shell to abort.

## Context

**Behavior:** Exit status rules for the `trap` special built-in:

1. **Invalid signal name or number:** A non-zero exit status shall be returned. [XSI] extends this to signal numbers in addition to names.
2. **Valid signal name/number:** Zero shall be returned.
3. **Critical non-abort rule:** For BOTH interactive AND non-interactive shells, invalid signal names (and [XSI] numbers) shall NOT be considered a shell error and shall NOT cause the shell to abort. This is an explicit exception to the general rule that special built-in errors cause non-interactive shells to exit.

**Applies to:** All shell modes (interactive and non-interactive).

**Implementation detail:** When `trap` receives an invalid signal name like `trap '' BOGUS`, the shell must:
- Return non-zero exit status.
- Write a diagnostic to stderr (recommended but not strictly required by this clause).
- Continue execution — do NOT abort, even in a non-interactive shell.

This is notable because most special built-in errors DO cause non-interactive shells to abort. The `trap` built-in has an explicit exception.

**[XSI] extension:** The XSI option adds support for signal numbers (in addition to names) as operands to `trap`.

**Spec reference:** Section 2.29 "trap" EXIT STATUS, lines 5611-5615 in `utilities/V3_chap02.html`.
