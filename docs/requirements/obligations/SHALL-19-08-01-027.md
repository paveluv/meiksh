## Extracted Text

> In all of the cases shown in the table where an interactive shell is required not to exit and a non-interactive shell is required to exit, an interactive shell shall not perform any further processing of the command in which the error occurred.

## Context

Requires an interactive shell to stop processing the current command after an error, even though it does not exit. See 2.8.1 Consequences of Shell Errors in utilities/V3_chap02.html.

**Behavior required:**
- In all table cases where an interactive shell is required NOT to exit and a non-interactive shell IS required to exit, the interactive shell SHALL NOT perform any further processing of the command in which the error occurred.
- This means: the interactive shell aborts the erroneous command entirely (no partial execution) and returns to the prompt.

**Error categories this applies to:**
- Syntax errors (interactive: don't exit, non-interactive: exit)
- Special built-in errors (interactive: don't exit, non-interactive: exit)
- Redirection errors with special built-ins (interactive: don't exit, non-interactive: exit)
- Variable assignment errors (interactive: don't exit, non-interactive: exit)
- Expansion errors (interactive: don't exit, non-interactive: exit)

**Implementation note:** After detecting one of these errors in interactive mode, the shell must discard any remaining parts of the current command (e.g., in a pipeline or compound command) and return to the read-eval loop.
