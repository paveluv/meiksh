## Extracted Text

> When there is no longer any process whose process ID or process group ID matches the foreground process group ID, the terminal shall have no foreground process group. It is unspecified whether the terminal has a foreground process group when there is a process whose process ID matches the foreground process group ID, but whose process group ID does not. No actions defined in POSIX.1-2024, other than allocation of a controlling terminal or a successful call to tcsetpgrp(), shall cause a process group to become the foreground process group of the terminal.

## Context

**Behavior:** When all processes matching the foreground process group ID have exited, the terminal shall have no foreground process group. Only controlling terminal allocation or tcsetpgrp() can set a new foreground process group — no other POSIX action does this.

**Conditions:** This matters when foreground jobs terminate. The shell must call tcsetpgrp() to reassert its own process group as the foreground group after a foreground job completes.

**Edge case:** If a process's PID matches the foreground PGID but its actual PGID differs, whether the terminal has a foreground group is unspecified.

**Shell relevance:** After a foreground job exits, the shell must explicitly call tcsetpgrp() to reclaim the terminal. The terminal does not automatically reassign the foreground group.
