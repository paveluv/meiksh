## Extracted Text

> If a \e or \cX escape sequence specifies a character that does not have an encoding in the locale in effect when these <backslash>-escape sequences are processed, the result is implementation-defined. However, implementations shall not replace an unsupported character with bytes that do not form valid characters in that locale's character set.

## Context

**Behavior:** Two related rules:

1. **Implementation-defined result:** If a `\e` or `\cX` escape sequence specifies a character that does not have an encoding in the current locale (the locale in effect when escape processing occurs), the result is implementation-defined. The implementation must document what it does in this case.

2. **Mandatory constraint:** Regardless of the implementation-defined choice, the implementation MUST NOT replace an unsupported character with bytes that do not form valid characters in that locale's character set. In other words, the output must always be valid in the current locale's encoding—no broken/incomplete multibyte sequences.

**Applies to:** All shell modes during `$'...'` escape processing.

**Implementation detail:** For ASCII/UTF-8 locales, ESC (0x1B) and all control characters have valid encodings, so this issue typically doesn't arise. It may matter in rare non-ASCII locales (e.g., some EBCDIC variants). A safe fallback is to emit an empty string or a locale-appropriate replacement character.
