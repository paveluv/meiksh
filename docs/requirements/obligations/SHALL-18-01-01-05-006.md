## Extracted Text

> When a directory that is the root directory or current working directory of any process is removed, the effect is implementation-defined. If file access permissions deny access, the requested operation shall fail. Otherwise, when a file is removed:: If the file is not a directory: If the link count becomes zero: If no process has the file open, the space occupied by the file shall be freed and the file shall no longer be accessible. If one or more processes have the file open, the file contents shall be preserved until all references to the file have been closed. If the link count is not reduced to zero, the last file status change timestamp shall be marked for update.

## Context

If the removed file is not a directory: (a) if the link count becomes zero: (i) if no process has the file open, the space is freed and the file is no longer accessible; (ii) if processes have the file open, the contents are preserved until all references are closed. (b) If the link count is not reduced to zero, the status change timestamp is marked for update. For `meiksh`, this is important because the shell may hold file descriptors to files that are unlinked: here-document temp files, for instance, may be unlinked immediately after opening. The file data remains accessible through the open file descriptor until it is closed. This is a common pattern for secure temporary files. See POSIX Utilities Chapter 1, Section 1.1.1.5 item 5 (utilities/V3_chap01.html lines 711–721).
