## Extracted Text

> A pathname consisting of a single <slash> shall resolve to the root directory of the process. A null pathname shall not be successfully resolved. If a pathname begins with two successive <slash> characters, the first component following the leading <slash> characters may be interpreted in an implementation-defined manner, although more than two leading <slash> characters shall be treated as a single <slash> character.

## Context

Three critical pathname resolution rules for the shell:

1. **Single slash `/`**: Resolves to the root directory. The shell must handle `cd /`, `test -d /`, etc. correctly.
2. **Null/empty pathname**: Must fail. The shell must not allow empty strings as pathnames in redirections, `cd`, command execution, etc. An empty `$var` used as a pathname should cause an error.
3. **Double-slash `//` prefix**: The first component after `//` may have implementation-defined meaning (some systems use `//host/share` for network paths). The shell's `cd -L` logical path canonicalization must preserve a leading `//` (but not `///` or more). Specifically:
   - `//foo` — the `//` prefix is preserved (implementation-defined interpretation).
   - `///foo` or more — treated as a single `/`, so `/foo`.

This directly affects the shell's `cd` builtin path canonicalization algorithm and `PWD` tracking.
