## Extracted Text

> where expression consists of all characters until the matching '}'. Any '}' escaped by a <backslash> or within a quoted string, and characters in embedded arithmetic expansions, command substitutions, and variable expansions, shall not be examined in determining the matching '}'.

## Context

**Behavior:** When parsing a braced parameter expansion `${expression}`, the shell must find the matching closing `}` using proper nesting rules. A `}` character must NOT be treated as the closing delimiter if it:
1. Is escaped by a preceding `\` (backslash)
2. Appears within a single-quoted or double-quoted string inside the expansion
3. Appears within a nested arithmetic expansion `$(( ... ))`
4. Appears within a nested command substitution `$( ... )` or `` ` ... ` ``
5. Appears within a nested variable expansion `${ ... }`

**Conditions:** Applies during the tokenization/parsing phase whenever `${` is encountered. The parser must track quoting context and nesting depth of embedded expansions.

**Inputs/Outputs:**
- Input: source text starting with `${`
- Output: correctly identified span of the parameter expansion, with `expression` being everything up to the true matching `}`

**Error handling:** If no matching `}` is found (unterminated expansion), a syntax error should be raised.

**Spec reference:** See 2.6.2 Parameter Expansion in utilities/V3_chap02.html, lines 778–783. Also see 2.2.3 Double-Quotes for quoting interaction rules.
