## Extracted Text

> For a parameter that is not enclosed in braces:: If the parameter is a name, the expansion shall use the longest valid name (see XBD 3.216 Name), whether or not the variable denoted by that name exists.

## Context

**Behavior:** When parsing an unbraced parameter expansion (e.g., `$varname`), the shell must use "longest match" to determine the parameter name. A "name" consists of underscores, digits, and letters, starting with an underscore or letter (per XBD 3.216 Name). The shell must consume as many valid name characters as possible, even if the resulting variable is not set.

**Conditions:** Applies only to unbraced expansions (`$name`, not `${name}`). This is a parsing rule — the shell decides how many characters form the parameter name before evaluating whether the variable exists.

**Inputs/Outputs:**
- Input: `$` followed by a character that starts a valid name
- Output: the longest sequence of valid name characters is consumed as the parameter name. For example, in `$abc123def`, the entire `abc123def` is the parameter name (if all characters are valid name characters), regardless of whether `abc123def` is defined as a variable.

**Error handling:** No specific error; if the resulting name refers to an unset variable, normal unset-variable rules apply (empty substitution, or error under `set -u`).

**Spec reference:** See 2.6.2 Parameter Expansion in utilities/V3_chap02.html, lines 794–795. For the definition of "Name", see basedefs/V1_chap03.html#tag_03_216.
