## Extracted Text

> If a process is in the foreground process group of its controlling terminal, read operations shall be allowed, as described in 11.1.5 Input Processing and Reading Data. Any attempts by a process in a background process group to read from its controlling terminal cause its process group to be sent a SIGTTIN signal unless one of the following special cases applies: if the reading process is ignoring the SIGTTIN signal or the reading thread is blocking the SIGTTIN signal, or if the process group of the reading process is orphaned, the read() shall return -1, with errno set to [EIO] and no signal shall be sent. The default action of the SIGTTIN signal shall be to stop the process to which it is sent. See <signal.h>.

## Context

**Behavior:** Terminal read access control:
- Foreground process group → reads are allowed normally
- Background process group attempting to read → SIGTTIN signal sent to the process group (default: stop)
- Exceptions where SIGTTIN is NOT sent (read returns -1/EIO instead):
  - Reading process is ignoring SIGTTIN
  - Reading thread is blocking SIGTTIN
  - Process group is orphaned

**Error handling:** When exceptions apply, read() returns -1 with errno=EIO.

**Shell relevance:** Critical for job control. Background jobs that try to read from the terminal are stopped by SIGTTIN. The shell must handle SIGTTIN correctly: it should ignore SIGTTIN itself (to avoid being stopped) and must be prepared for stopped background jobs to be resumed via `fg`.
