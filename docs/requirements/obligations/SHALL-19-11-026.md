## Extracted Text

> If a process that the shell is waiting for is part of a foreground job that was started as a foreground job and is stopped by a SIGSTOP signal, the behavior shall be as described above for a catchable signal unless the shell was executing a built-in utility in the current shell execution environment when the SIGSTOP was delivered, resulting in the shell itself being stopped by the signal, in which case if the shell subsequently receives a SIGCONT signal and has one or more child processes that remain stopped, the shell shall create a suspended job as if only those child processes had been stopped.

## Context

**SIGSTOP handling in foreground jobs:** SIGSTOP is non-catchable (cannot be trapped), unlike SIGTSTP/SIGTTIN/SIGTTOU. When a foreground job process is stopped by SIGSTOP, the behavior is generally the same as for catchable stop signals (create a suspended job per SHALL-19-11-022/023), **except**:

If the shell was executing a builtin in the current shell environment when SIGSTOP was delivered (causing the shell process itself to be stopped), and the shell subsequently receives SIGCONT, and it has one or more child processes that remain stopped, the shell shall create a suspended job as if only those child processes had been stopped.

This handles the edge case where SIGSTOP stops the shell process itself (since it can't be caught). Upon SIGCONT, the shell recovers by checking for stopped children and creating appropriate suspended jobs.
