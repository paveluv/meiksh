## Extracted Text

> When jobs reports the termination status of a job, the shell shall remove the job from the background jobs list and the associated process ID from the list of those "known in the current shell execution environment"; see 2.9.3.1 Asynchronous AND-OR Lists. If a write error occurs when jobs writes to standard output, some process IDs might have been removed from the list but not successfully reported.

## Context

**Behavior:** When `jobs` reports that a job has terminated (Done, Done(code), or signal-terminated), the shell must:
1. Remove the job from the background jobs list.
2. Remove the associated process ID from the "known PIDs" list (used by `$!` and `wait`).

**Write error caveat:** If a write error occurs while `jobs` is writing to stdout, some jobs may have already been removed from the internal list but their status was not successfully displayed. This is explicitly allowed — the shell does not need to roll back list removals on write failure.

**Cross-reference:** See Section 2.9.3.1 for the definition of "known in the current shell execution environment" process ID list.

**Spec reference:** POSIX.1-2024, `jobs` DESCRIPTION, lines 42-45 of `utilities/jobs.html`.
