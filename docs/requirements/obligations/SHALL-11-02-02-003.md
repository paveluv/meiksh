## Extracted Text

> If IGNBRK is set, a break condition detected on input shall be ignored; that is, not put on the input queue and therefore not read by any process. If IGNBRK is not set and BRKINT is set, the break condition shall flush the input and output queues, and if the terminal is the controlling terminal of a foreground process group, the break condition shall generate a single SIGINT signal to that foreground process group. If neither IGNBRK nor BRKINT is set, a break condition shall be read as a single 0x00, or if PARMRK is set, as 0xff 0x00 0x00.

## Context

**Behavior:** Break condition handling depends on IGNBRK, BRKINT, and PARMRK flags:
1. **IGNBRK set**: break is silently ignored (not queued, not readable).
2. **IGNBRK not set, BRKINT set**: break flushes input and output queues, and if the terminal is the controlling terminal, generates a single SIGINT to the foreground process group.
3. **Neither IGNBRK nor BRKINT set**: break is read as a single 0x00 byte. If PARMRK is also set, it is read as the three-byte sequence 0xff 0x00 0x00.

**Conditions:** Applies to terminal input when a break condition is detected. These flags are in `c_iflag`.

**Shell relevance:** The shell should set BRKINT for interactive use so that a break condition (if received) acts like an interrupt. The `stty` builtin must support `ignbrk`, `brkint`, and `parmrk` flag settings. For the `read` builtin, break behavior follows these rules if the terminal has the appropriate flags set.
