## Extracted Text

> The following options shall be supported:: (The letter ell.) Write all values of signal_name supported by the implementation, if no operand is given. If an exit_status operand is given and it is a value of the '?' shell special parameter (see 2.5.2 Special Parameters and wait) corresponding to a process that was terminated or stopped by a signal, the signal_name corresponding to the signal that terminated or stopped the process shall be written. If an exit_status operand is given and it is the unsigned decimal integer value of a signal number, the signal_name (the symbolic constant name without the SIG prefix defined in the Base Definitions volume of POSIX.1-2024) corresponding to that signal shall be written. Otherwise, the results are unspecified.

## Context

**Behavior:** The `-l` option has three modes:
1. **No operand:** Write all signal names supported by the implementation to stdout (without the `SIG` prefix, in uppercase).
2. **exit_status is a `$?` value from a signal-killed/stopped process:** Write the signal name that terminated or stopped that process. The shell encodes signal exits as values > 128; `kill -l $?` must reverse this mapping.
3. **exit_status is an unsigned decimal signal number:** Write the corresponding signal name (without `SIG` prefix).
4. **Otherwise:** Results are unspecified (e.g., exit_status is not a valid signal number and not a signal-based exit status).

**Inputs:** Optional `exit_status` operand — a decimal integer.

**Outputs:** Signal name(s) written to stdout per the format in SHALL-20-64-10-002 through SHALL-20-64-10-004.

**Error handling:** If the exit_status doesn't match any known signal number or signal-based exit status, behavior is unspecified — the implementation may write nothing, write an error, or produce any output.

**Spec reference:** Section OPTIONS (tag_20_64_04) in `utilities/kill.html`, lines 69–75. See also XCU 2.5.2 Special Parameters for `$?` semantics.
