## Extracted Text

> A sequence of bytes consisting of 1 to {NAME_MAX} bytes used to name a file. The bytes composing the name shall not contain the <NUL> or <slash> characters. In the context of a pathname, each filename shall be followed by a <slash> or a <NUL> character; elsewhere, a filename followed by a <NUL> character forms a string (but not necessarily a character string). The filenames dot and dot-dot have special meaning. A filename is sometimes referred to as a "pathname component". See also 3.254 Pathname.

## Context

**Requirement — direct shell relevance.** Defines "filename" (pathname component) as a sequence of 1 to `{NAME_MAX}` bytes. Two mandatory constraints:
1. A filename SHALL NOT contain NUL (`\0`) bytes
2. A filename SHALL NOT contain slash (`/`) bytes

In pathname context, each filename is followed by `/` or NUL. The filenames `.` (dot) and `..` (dot-dot) have special meaning.

**Shell impact:** The shell encounters filenames during:
- Pathname expansion (globbing): generated filenames must conform to these rules
- `cd` builtin: parses pathnames into components
- Command lookup in `$PATH`: splits path and appends command names
- Here-document temp files, process substitution FIFOs

The shell must never generate filenames containing NUL or `/` in a component. When performing pathname expansion, the OS ensures returned filenames are valid, but the shell must handle the `.` and `..` special names correctly (e.g., not matching `.*` glob unexpectedly in some contexts).
