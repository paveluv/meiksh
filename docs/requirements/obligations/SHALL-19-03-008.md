## Extracted Text

> If the current character is an unquoted '$' or '`', the shell shall identify the start of any candidates for parameter expansion ( 2.6.2 Parameter Expansion), command substitution ( 2.6.3 Command Substitution), or arithmetic expansion ( 2.6.4 Arithmetic Expansion) from their introductory unquoted character sequences: '$' or "${", "$(" or '`', and "$((", respectively. The shell shall read sufficient input to determine the end of the unit to be expanded (as explained in the cited sections). While processing the characters, if instances of expansions or quoting are found nested within the substitution, the shell shall recursively process them in the manner specified for the construct that is found. For "$(" and '`' only, if instances of io_here tokens are found nested within the substitution, they shall be parsed according to the rules of 2.7.4 Here-Document; if the terminating ')' or '`' of the substitution occurs before the NEWLINE token marking the start of the here-document, the behavior is unspecified. The characters found from the beginning of the substitution to its end, allowing for any recursion necessary to recognize embedded constructs, shall be included unmodified in the result token, including any embedded or enclosing substitution operators or quotes. The token shall not be delimited by the end of the substitution.

## Context

**Behavior:** Token recognition Rule 5: If the current character is an unquoted `$` or `` ` ``, the shell must identify the start of expansion candidates and read sufficient input to find the end of the construct. Key requirements:

1. **Expansion candidate identification:**
   - `$` or `${` → parameter expansion (2.6.2)
   - `$(` or `` ` `` → command substitution (2.6.3)
   - `$((` → arithmetic expansion (2.6.4)

2. **Read to end of construct:** The shell reads enough input to find the matching `)`, `` ` ``, or `}`. For `$(...)`, matching uses recursive tokenization. For `` `...` ``, matching finds the next unescaped `` ` ``. For `${...}`, matching finds `}` accounting for nesting.

3. **Recursive processing:** Nested expansions and quoting within the construct are processed recursively—the shell must handle arbitrary nesting depth.

4. **Here-documents in command substitution:** For `$(...)` and `` `...` `` only, `io_here` tokens inside are parsed per 2.7.4. If the closing `)` or `` ` `` occurs before the NEWLINE starting the here-document, behavior is unspecified.

5. **Token preservation:** All characters from the start to the end of the construct (including nested quotes, operators, etc.) are included unmodified in the result token.

6. **No delimiting:** The end of the expansion construct does NOT delimit the token.

**Applies to:** All shell modes during tokenization.

**Implementation detail:** This is one of the most complex tokenizer rules. The tokenizer must recursively enter sub-tokenization contexts for `$(...)` and `$((...))`—essentially running the full tokenizer/parser inside. For `` `...` ``, simpler backquote matching suffices (with `\` escaping only `$`, `` ` ``, `\`).

**Spec reference:** Section 2.3, rule 5, lines 275-285 in `utilities/V3_chap02.html`. Cross-refs: 2.6.2, 2.6.3, 2.6.4, 2.7.4.
