## Extracted Text

> The following options shall be supported:: [XSI] Specify a non-negative decimal integer, signal_number, representing the signal to be used instead of SIGTERM, as the sig argument in the effective call to kill(). The correspondence between integer values and the sig value used is shown in the following list. The effects of specifying any signal_number other than those listed below are undefined. 0 0 1 SIGHUP 2 SIGINT 3 SIGQUIT 6 SIGABRT 9 SIGKILL 14 SIGALRM 15 SIGTERM If the first argument is a negative integer, it shall be interpreted as a -signal_number option, not as a negative pid operand specifying a process group.

## Context

**Behavior:** The `-signal_number` option (XSI) specifies the signal by number. The mandatory mapping is:
- `0` → signal 0 (existence test)
- `1` → SIGHUP
- `2` → SIGINT
- `3` → SIGQUIT
- `6` → SIGABRT
- `9` → SIGKILL
- `14` → SIGALRM
- `15` → SIGTERM

The effects of specifying any signal_number other than those listed above are **undefined** (implementation may support additional numbers but is not required to).

**Disambiguation rule:** If the first argument to `kill` is a negative integer (e.g., `kill -9 ...`), it must always be interpreted as `-signal_number`, never as a negative pid operand specifying a process group. Users must use `--` to separate options from negative pid operands (e.g., `kill -- -123`).

**Inputs:** Non-negative decimal integer as the signal_number.
