## Extracted Text

> In order to avoid data races, applications shall ensure that non-lock-free access to any memory location by more than one thread of control (threads or processes) is restricted such that no thread of control can read or modify a memory location while another thread of control might be modifying it. ## Context

This is a C/pthreads-level requirement about memory synchronization functions. For the shell implementation (written in Rust), this is handled by Rust's ownership and borrowing system, `Mutex`, `RwLock`, and atomic types. The shell does not expose threading to shell scripts — POSIX shell is single-threaded from the user's perspective (concurrency is via subprocesses, not threads). This obligation constrains the implementation language's use of synchronization primitives internally. No shell-language-level action is required.
