## Extracted Text

> If the previous character was used as part of an operator and the current character is not quoted and can be used with the previous characters to form an operator, it shall be used as part of that (operator) token.

## Context

**Behavior:** Token recognition Rule 2: If the previous character was part of an operator token being built, and the current character is unquoted AND can extend that operator to form a longer valid operator, then the current character becomes part of the operator token.

**Applies to:** All shell modes during tokenization.

**Multi-character operators in the shell grammar:** `&&`, `||`, `;;`, `<<`, `>>`, `<&`, `>&`, `<>`, `<<-`, `>|`, `;;`, and `NEWLINE`. The rule handles building these character-by-character.

**Implementation detail:** The tokenizer must know which operator prefixes can be extended. For example:
- After `<`, the characters `<`, `&`, `>` can extend it to `<<`, `<&`, `<>` respectively.
- After `<<`, only `-` can extend it to `<<-`.
- After `>`, the characters `>`, `&`, `|` can extend it.
- After `&` or `|`, only the same character extends it (`&&`, `||`).
- After `;`, only `;` extends it (`;;`).

If the current character can extend, it is consumed into the operator token. If not, rule 3 applies.

**Spec reference:** Section 2.3, rule 2, lines 259-260 in `utilities/V3_chap02.html`.
