## Extracted Text

> In locales other than the POSIX locale, a character may have a state-dependent encoding. There are two types of these encodings:: A single-shift encoding (where each character not in the initial shift state is preceded by a shift code) can be defined if each shift-code and character sequence is considered a multi-byte character. This is done using the concatenated-constant format in a character set description file, as described in 6.4 Character Set Description File. If the implementation supports a character encoding of this type, all of the standard utilities in the Shell and Utilities volume of POSIX.1-2024 shall support it. Use of a single-shift encoding with any of the functions in the System Interfaces volume of POSIX.1-2024 that do not specifically mention the effects of state-dependent encoding is implementation-defined.

## Context

If the system supports single-shift character encodings (where shift codes are treated as part of a multibyte character), the shell and all its builtins MUST support them. This means:

1. **Pattern matching**: `case`, `[[ ]]`, and pathname expansion must handle single-shift encoded characters correctly.
2. **String operations**: `${#var}` must count characters (not bytes), `${var#pattern}` and similar must operate on character boundaries.
3. **`read` builtin**: Must read complete multibyte characters including shift codes.
4. **Field splitting**: IFS processing must not split in the middle of a multibyte character.

In practice, this requirement is satisfied by correct multibyte/wide-character handling (using `mbrtowc()`, `wcwidth()`, etc.).

See: `basedefs/V1_chap06.html` §6.2 Character Encoding; §6.4 Character Set Description File.
