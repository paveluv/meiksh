## Extracted Text

> If the system detects a loop in the pathname resolution process, pathname resolution shall fail with functions reporting an [ELOOP] error and utilities writing an equivalent diagnostic message. The same may happen if during the resolution process more symbolic links were followed than the implementation allows. This implementation-defined limit shall not be smaller than {SYMLOOP_MAX}.

## Context

Symlink loops (e.g., `a -> b`, `b -> a`) must be detected and reported as ELOOP errors. The OS kernel handles this detection, but the shell must:

1. **Propagate ELOOP errors**: When any shell operation (command execution, `cd`, redirection, `.` sourcing, `test` file operators) encounters ELOOP, the shell must write an appropriate diagnostic message to stderr.
2. **Respect SYMLOOP_MAX**: The implementation-defined limit on symlink traversals must be at least `{SYMLOOP_MAX}` (typically 8 or more). The shell doesn't control this limit — it's a kernel parameter.

This is particularly relevant for `cd -L` which may follow long chains of symlinks, and for pathname expansion patterns that traverse symlinked directory trees.

See: `basedefs/V1_chap04.html` §4.16 Pathname Resolution.
