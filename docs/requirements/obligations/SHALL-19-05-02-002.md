## Extracted Text

> Expands to the positional parameters, starting from one, initially producing one field for each positional parameter that is set. When the expansion occurs in a context where field splitting will be performed, any empty fields may be discarded and each of the non-empty fields shall be further split as described in 2.6.5 Field Splitting. When the expansion occurs within double-quotes, the behavior is unspecified unless one of the following is true: Field splitting as described in 2.6.5 Field Splitting would be performed if the expansion were not within double-quotes (regardless of whether field splitting would have any effect; for example, if IFS is null). The double-quotes are within the word of a ${parameter:-word} or a ${parameter:+word} expansion (with or without the <colon>; see 2.6.2 Parameter Expansion) which would have been subject to field splitting if parameter had been expanded instead of word. If one of these conditions is true, the initial fields shall be retained as separate fields, except that if the parameter being expanded was embedded within a word, the first field shall be joined with the beginning part of the original word and the last field shall be joined with the end part of the original word. In all other contexts the results of the expansion are unspecified. If there are no positional parameters, the expansion of '@' shall generate zero fields, even when '@' is within double-quotes; however, if the expansion is embedded within a word which contains one or more other parts that expand to a quoted null string, these null string(s) shall still produce an empty field, except that if the other parts are all within the same double-quotes as the '@', it is unspecified whether the result is zero fields or one empty field.

## Context

This defines the expansion behavior of the special parameter `$@` (at-sign).

**Required behavior — outside double-quotes:**
- `$@` expands to the positional parameters starting from `$1`, initially producing one field per set positional parameter.
- In a field-splitting context, empty fields may be discarded, and each non-empty field is further split according to IFS (see 2.6.5 Field Splitting).

**Required behavior — inside double-quotes (`"$@"`):**
The behavior is unspecified UNLESS one of two conditions holds:
1. Field splitting (per 2.6.5) *would* be performed if the expansion were not within double-quotes (regardless of whether IFS would actually split anything — e.g., even if IFS is null).
2. The double-quotes are within the `word` part of `${parameter:-word}` or `${parameter:+word}` (with or without colon) that would have been subject to field splitting if `parameter` had been expanded instead of `word`.

When one of those conditions is true:
- The initial fields (one per positional parameter) are retained as separate fields.
- If `$@` is embedded within a larger word, the first field is joined with the text before `$@` and the last field is joined with the text after `$@`.

**Zero positional parameters:**
- `"$@"` must generate zero fields (not one empty field).
- However, if `"$@"` is embedded in a word that contains other parts expanding to quoted null strings, those null strings still produce an empty field — UNLESS all the other parts are within the same double-quotes as `@`, in which case the result (zero fields vs. one empty field) is unspecified.

**Conditions:** All execution contexts. The distinction between field-splitting and non-field-splitting contexts is critical (e.g., assignment `var=$@` is a non-splitting context).

See 2.5.2 "Special Parameters" (`@`), 2.6.2 "Parameter Expansion", and 2.6.5 "Field Splitting" in `utilities/V3_chap02.html`.
