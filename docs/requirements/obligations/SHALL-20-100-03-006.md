## Extracted Text

> If IFS is unset, or is set to any non-empty value, then a modified version of the field splitting algorithm specified in 2.6.5 Field Splitting shall be applied, with the modifications as follows:: The input to the algorithm shall be the logical line (minus terminating delimiter) that was read from standard input, and shall be considered as a single initial field, all of which resulted from expansions, with any escaped byte and the preceding <backslash> escape character treated as if they were the result of a quoted expansion, and all other bytes treated as if they were the results of unquoted expansions.

## Context

**Behavior required:** The input to the modified field splitting algorithm is the logical line (with the terminating delimiter removed), treated as a single initial field with these properties:

1. **Escaped bytes** (a byte preceded by an unescaped backslash, when `-r` is not specified): treated as if they were the result of a quoted expansion. This means IFS characters that are escaped are NOT used as field separators.
2. **All other bytes**: treated as if they were the result of unquoted expansions. This means unescaped IFS characters ARE used as field separators.

**Condition:** When `IFS` is unset or non-empty, during the field splitting phase of `read`.

**Implementation detail:** When implementing field splitting for `read`:
- Track which bytes in the input are "escaped" (preceded by backslash when `-r` is not in effect)
- Escaped bytes and their preceding backslash are treated like quoted text (immune to field splitting)
- Everything else is subject to normal IFS splitting rules
- The entire input line is one "field" to start with; the splitting algorithm then breaks it apart
