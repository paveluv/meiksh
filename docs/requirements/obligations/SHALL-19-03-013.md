## Extracted Text

> In situations where the shell parses its input as a program, once a complete_command has been recognized by the grammar (see 2.10 Shell Grammar), the complete_command shall be executed before the next complete_command is tokenized and parsed.

## Context

**Behavior:** When the shell parses input as a program (as opposed to processing here-documents or other non-program contexts), each `complete_command` recognized by the grammar (see 2.10 Shell Grammar) must be executed BEFORE the next `complete_command` is tokenized and parsed.

**Applies to:** Both interactive and non-interactive shell modes.

**What is a `complete_command`:** Per the grammar, a `complete_command` is a `list` optionally followed by `;` or `&`, terminated by a NEWLINE or end-of-input. In interactive mode, each line typically forms one `complete_command`. In scripts, they may span multiple lines (e.g., `if...fi`, `while...done`).

**Why this matters:** This ensures that side effects of one command (e.g., `alias`, `export`, variable assignments, `cd`, function definitions) are visible to the parsing/tokenization of subsequent commands. Without this rule, a shell could read-ahead and tokenize future commands before executing earlier ones, potentially missing alias definitions or variable changes that affect tokenization.

**Implementation detail:** The shell must not pipeline tokenization/parsing ahead of execution. After the parser produces a complete AST for a `complete_command`, it must hand it to the executor, wait for completion, then resume tokenization for the next `complete_command`.

**Spec reference:** Section 2.3, lines 310-312 in `utilities/V3_chap02.html`. Cross-ref: 2.10 Shell Grammar.
