## Extracted Text

> If no readable file was found or if the commands in the file could not be parsed, and the shell is interactive (and therefore does not abort; see 2.8.1 Consequences of Shell Errors), the exit status shall be non-zero. Otherwise, return the value of the last command executed, or a zero exit status if no command is executed.

## Context

**Behavior:** Exit status rules for the `dot` (`.`) special built-in:

1. **Error cases (interactive shell):** If no readable file was found, or the commands in the file could not be parsed, AND the shell is interactive (which means it does not abort on such errors per 2.8.1), then the exit status shall be non-zero.
2. **Error cases (non-interactive shell):** A non-interactive shell shall abort (exit) when dot cannot find a readable file or encounters a parse error — the exit status rules here don't apply because the shell terminates per 2.8.1 Consequences of Shell Errors.
3. **Success cases:** If the file is found, readable, and parseable, the exit status shall be the exit status of the last command executed in the sourced file.
4. **Empty file:** If the file is found and readable but contains no commands (only blank lines or comments), the exit status shall be zero.

**Applies to:** All shell modes, but the non-zero-on-error clause specifically applies to interactive shells (non-interactive shells abort instead).

**Cross-reference:** See 2.8.1 Consequences of Shell Errors (`utilities/V3_chap02.html#tag_19_08_01`) for the abort behavior in non-interactive shells.
