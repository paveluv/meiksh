## Extracted Text

> If PARMRK is set, and IGNPAR is not set, a byte with a framing or parity error (other than break) shall be given to the application as the three-byte sequence 0xff 0x00 X, where 0xff 0x00 is a two-byte flag preceding each sequence and X is the data of the byte received in error. To avoid ambiguity in this case, if ISTRIP is not set, a valid byte of 0xff is given to the application as 0xff 0xff. If neither PARMRK nor IGNPAR is set, a framing or parity error (other than break) shall be given to the application as a single byte 0x00.

## Context

**Behavior:** Parity/framing error marking depends on PARMRK, IGNPAR, and ISTRIP:
1. **PARMRK set, IGNPAR not set**: error bytes are delivered as the three-byte sequence `0xff 0x00 X` where X is the errored byte. To avoid ambiguity, valid 0xff bytes are delivered as `0xff 0xff` (when ISTRIP is not set).
2. **Neither PARMRK nor IGNPAR set**: error bytes are delivered as a single `0x00`.
3. Break conditions are handled separately (see SHALL-11-02-02-003).

**Conditions:** Applies to terminal input. PARMRK and IGNPAR are in `c_iflag`. ISTRIP affects the ambiguity handling.

**Shell relevance:** Primarily relevant for serial terminals. The `stty` builtin must support `parmrk`/`-parmrk`. The shell should be aware that with PARMRK set, 0xff bytes in input may be doubled, which could affect binary data processing. For normal interactive use on PTYs, this is rarely relevant.

**Spec reference:** POSIX.1-2024, §11.2.2 Input Modes (tag_11_02_02), `basedefs/V1_chap11.html` line 511.
