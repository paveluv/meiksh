## Extracted Text

> The DESCRIPTION section describes the actions of the utility. If the utility has a very complex set of subcommands or its own procedural language, an EXTENDED DESCRIPTION section is also provided. Most explanations of optional functionality are omitted here, as they are usually explained in the OPTIONS section. As stated in 1.1.1.11 Actions Equivalent to Functions, some functions are described in terms of equivalent functionality. When specific functions are cited, the implementation shall provide equivalent functionality including side-effects associated with successful execution of the function. The treatment of errors and intermediate results from the individual functions cited is generally not specified by this volume of POSIX.1-2024. See the utility's EXIT STATUS and CONSEQUENCES OF ERRORS sections for all actions associated with errors encountered by the utility. A standard utility shall not be treated as a declaration utility unless explicitly stated in this section.

## Context

When implementing shell builtins, the shell **shall** provide equivalent functionality to any cited system functions, including their side effects on successful execution. Error handling and intermediate results from cited functions are generally not specified — see each builtin's EXIT STATUS and CONSEQUENCES OF ERRORS sections instead. A standard utility **shall not** be treated as a declaration utility (one that declares variables/functions in the calling scope, like `export`, `readonly`, `typeset`) unless the DESCRIPTION section explicitly says so. This affects how the shell processes command prefixes and assignments for builtins. Reference: XCU 1.4, XCU 1.1.1.11.
