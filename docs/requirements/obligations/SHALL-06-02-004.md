## Extracted Text

> While in the initial shift state, all characters in the portable character set shall retain their usual interpretation and shall not alter the shift state. The interpretation for subsequent bytes in the sequence shall be a function of the current shift state. A byte with all bits zero shall be interpreted as the null character independent of shift state. Such a byte shall not occur as part of any other character. Likewise, the byte values used to encode <period>, <slash>, <newline>, and <carriage-return> shall not occur as part of any other character in any locale.

## Context

This is a critical encoding safety guarantee for the shell:

1. **Portable character set in initial state**: All shell syntax characters (which are in the portable character set) retain their meaning in the initial shift state. The shell can always recognize its syntax characters at the start of input or after returning to the initial shift state.
2. **NUL byte uniqueness**: The NUL byte (0x00) is always the null character, never part of another character. The shell can safely use NUL as a string terminator.
3. **Special byte value uniqueness**: The byte values for `.`, `/`, newline, and carriage return NEVER appear as part of another character's encoding. This is essential for the shell:
   - `/` can always be recognized as a path separator, even inside multibyte character sequences.
   - `\n` (newline) can always be recognized as a line terminator.
   - This guarantees that byte-level scanning for these characters is safe even in multibyte locales.

This property is satisfied by UTF-8 (where all ASCII bytes 0x00-0x7F only appear as themselves) and must be verified for any other supported encoding.
