## Extracted Text

> The <dollar-sign> shall retain its special meaning introducing parameter expansion (see 2.6.2 Parameter Expansion), a form of command substitution (see 2.6.3 Command Substitution), and arithmetic expansion (see 2.6.4 Arithmetic Expansion), but shall not retain its special meaning introducing the dollar-single-quotes form of quoting (see 2.2.4 Dollar-Single-Quotes). The input characters within the quoted string that are also enclosed between "$(" and the matching ')' shall not be affected by the double-quotes, but rather shall define the command(s) whose output replaces the "$(...)" when the word is expanded. The tokenizing rules in 2.3 Token Recognition shall be applied recursively to find the matching ')'. For the four varieties of parameter expansion that provide for substring processing (see 2.6.2 Parameter Expansion), within the string of characters from an enclosed "${" to the matching '}', the double-quotes within which the expansion occurs shall have no effect on the handling of any special characters. For parameter expansions other than the four varieties that provide for substring processing, within the string of characters from an enclosed "${" to the matching '}', the double-quotes within which the expansion occurs shall preserve the literal value of all characters, with the exception of the characters double-quote, backquote, <dollar-sign>, and <backslash>. If any unescaped double-quote characters occur within the string, other than in embedded command substitutions, the behavior is unspecified. The backquote and <dollar-sign> characters shall follow the same rules as for characters in double-quotes described in this section. The <backslash> character shall follow the same rules as for characters in double-quotes described in this section except that it shall additionally retain its special meaning as an escape character when followed by '}' and this shall prevent the escaped '}' from being considered when determining the matching '}' (using the rule in 2.6.2 Parameter Expansion).

## Context

**Behavior:** This is a duplicate extraction of the same spec text as SHALL-19-02-03-003 (lines 111-130), focusing on the `$` rules inside double-quotes. The comprehensive rules are:

1. **`$` introduces parameter expansion, command substitution, and arithmetic expansion** inside double-quotes, but does NOT introduce dollar-single-quotes (`$'`).
2. **`$(...)` creates a nested unquoted context** — the enclosing double-quotes do not affect characters inside. Tokenization rules (2.3) apply recursively to find the matching `)`.
3. **Substring parameter expansions** (`${param%pat}`, `${param%%pat}`, `${param#pat}`, `${param##pat}`): double-quotes have no effect on special characters between `${` and `}`.
4. **Other parameter expansions** (e.g., `${param:-word}`): double-quotes preserve literal values inside `${...}`, except for `"`, `` ` ``, `$`, and `\`.
5. **Unescaped `"` inside `${...}`** (outside embedded command substitutions): behavior is unspecified.
6. **`\` inside `${...}`** follows double-quote backslash rules, plus can escape `}` to prevent it from being the closing brace.

**Applies to:** All shell modes during word expansion.

**Spec reference:** Section 2.2.3, lines 111-130 in `utilities/V3_chap02.html`. Cross-refs: 2.6.2, 2.6.3, 2.6.4, 2.2.4, 2.3.
