## Extracted Text

> An implementation may defer the effect of a change to an alias but the change shall take effect no later than the completion of the currently executing complete_command (see 2.10 Shell Grammar). Changes to aliases shall not take effect out of order. Implementations may provide predefined aliases that are in effect when the shell is invoked.

## Context

**Behavior:** Three rules about alias lifecycle:

1. **Deferred effect:** An implementation may defer the effect of a change to an alias (via `alias` or `unalias`), but the change MUST take effect no later than the completion of the currently executing `complete_command`. This means within a single `complete_command`, alias changes might not be visible immediately, but they must be visible before the next `complete_command` starts.

2. **Ordering guarantee:** Changes to aliases must not take effect out of order. If alias `A` is defined, then alias `B` is defined, the definitions must become effective in that order.

3. **Predefined aliases:** Implementations may provide predefined aliases that are in effect when the shell is invoked (e.g., common aliases like `hash`, `type`, etc.).

**Applies to:** All shell modes.

**Implementation detail:** The simplest approach is to apply alias changes immediately (no deferral). If deferral is used, a flush point must exist at `complete_command` boundaries. The alias table should be ordered or timestamped to maintain ordering.
