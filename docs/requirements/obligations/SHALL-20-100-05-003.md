## Extracted Text

> The following operand shall be supported:: The name of an existing or nonexisting shell variable. If a var operand names the variable IFS , the behavior is unspecified. If a var operand names one of the variables LANG , LC_CTYPE , or LC_ALL and the new value assigned to the variable would change how the bytes in IFS form characters, or which characters in IFS are considered to be IFS white space (see 2.6.5 Field Splitting), it is unspecified what effects, if any, the change has on how read performs field splitting.

## Context

**Behavior required:**
1. **Variable existence:** The `var` operand can name either an existing or a nonexisting shell variable. If the variable doesn't exist, `read` creates it.
2. **IFS as var is unspecified:** If a var operand is literally `IFS` (e.g., `read IFS`), the behavior is unspecified. The implementation may allow it, ignore it, or produce an error.
3. **Locale variables as var:** If a var operand is `LANG`, `LC_CTYPE`, or `LC_ALL`, and the new value would change how IFS bytes form characters or which IFS characters count as IFS whitespace, the effects on field splitting within the same `read` invocation are unspecified.

**Condition:** When processing `var` operands for the `read` builtin.

**Implementation detail:** The implementation should:
- Accept any valid shell variable name
- Be aware that `read IFS` is unspecified territory (implementation may choose any behavior)
- Not rely on IFS-related behavior being consistent if the user reads into locale variables that change character interpretation
