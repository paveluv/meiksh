## Extracted Text

> Each field containing the results from an expansion shall be processed in order, intermixed with fields not containing the results of expansions, processed as described above, as if by using the following algorithm, examining bytes in the input field, from beginning to end:: When instructed to start the next iteration of the loop, this is the start of the loop. While the input (as modified by earlier iterations of this loop) is not empty: Consider the leading remaining byte or byte sequence of the input. No such byte sequence shall contain data such that some bytes in the sequence resulted from an expansion, and others did not, nor which contains bytes resulting from the results of more than one expansion. If the byte or sequence of bytes is: A byte (or sequence of bytes) in the input which did not result from an expansion: Append this byte (or sequence) to the candidate, and remove it from the input. Start the next iteration of the loop. A byte sequence in the input which resulted from an expansion and which does not form a character in IFS : Append the first byte of the sequence to the candidate, and remove that byte from the input. Start the next iteration of the loop. A byte sequence in the input which resulted from an expansion and which forms an IFS white space character: Remove that byte sequence from the input, consider the new leading input byte sequence, and repeat this step. A byte sequence in the input which resulted from an expansion and which forms an IFS character that is not IFS white space: Remove that byte sequence from the input, but note it was observed. At this point, if the candidate is not empty, or if a sequence of bytes representing an IFS character that is not IFS white space was seen at step 4, then a field is said to have been delimited, and the candidate shall become an output field. Empty (clear) the candidate, and start the next iteration of the loop.

## Context

**Behavior:** This is the core field splitting algorithm loop. For each leading byte/sequence in the input:

**Step 1 — Literal bytes (not from expansion):** Append to candidate, continue loop. Literal bytes are never treated as delimiters, regardless of IFS content.

**Step 2 — Expansion bytes, NOT an IFS character:** Append the first byte of the sequence to the candidate, remove that byte, continue loop. These are expansion-result bytes that don't match any IFS character.

**Step 3 — Expansion bytes forming IFS white space:** Remove the byte sequence from input and **repeat this step** (consume all consecutive IFS white space). IFS white space coalesces — multiple adjacent IFS-white-space characters from expansions count as a single delimiter boundary.

**Step 4 — Expansion bytes forming non-white-space IFS character:** Remove from input, note it was seen. Unlike IFS white space, each non-white-space IFS character is a distinct delimiter that can produce empty fields.

**Delimiting decision:** After consuming IFS characters (steps 3 & 4), if:
- The candidate is non-empty, OR
- A non-white-space IFS character was seen (step 4)

Then a field is delimited: the candidate becomes an output field, the candidate is cleared, and the loop continues.

**Key implications:**
- IFS white space coalesces and acts as a field terminator (no empty fields between runs of white space)
- Non-white-space IFS characters act as explicit separators (can produce empty fields: `IFS=:; x="a::b"` → `a`, ``, `b`)
- A non-white-space IFS char adjacent to IFS white space: the white space is consumed, the non-white-space char produces the field boundary

**Byte sequence constraint:** A byte sequence considered at each step must come entirely from one expansion or entirely from literal text — it cannot span expansion boundaries.
