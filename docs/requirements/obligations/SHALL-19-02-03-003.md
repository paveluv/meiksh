## Extracted Text

> Enclosing characters in double-quotes ("") shall preserve the literal value of all characters within the double-quotes, with the exception of the characters backquote, <dollar-sign>, and <backslash>, as follows:: The <dollar-sign> shall retain its special meaning introducing parameter expansion (see 2.6.2 Parameter Expansion), a form of command substitution (see 2.6.3 Command Substitution), and arithmetic expansion (see 2.6.4 Arithmetic Expansion), but shall not retain its special meaning introducing the dollar-single-quotes form of quoting (see 2.2.4 Dollar-Single-Quotes). The input characters within the quoted string that are also enclosed between "$(" and the matching ')' shall not be affected by the double-quotes, but rather shall define the command(s) whose output replaces the "$(...)" when the word is expanded. The tokenizing rules in 2.3 Token Recognition shall be applied recursively to find the matching ')'. For the four varieties of parameter expansion that provide for substring processing (see 2.6.2 Parameter Expansion), within the string of characters from an enclosed "${" to the matching '}', the double-quotes within which the expansion occurs shall have no effect on the handling of any special characters. For parameter expansions other than the four varieties that provide for substring processing, within the string of characters from an enclosed "${" to the matching '}', the double-quotes within which the expansion occurs shall preserve the literal value of all characters, with the exception of the characters double-quote, backquote, <dollar-sign>, and <backslash>. If any unescaped double-quote characters occur within the string, other than in embedded command substitutions, the behavior is unspecified. The backquote and <dollar-sign> characters shall follow the same rules as for characters in double-quotes described in this section. The <backslash> character shall follow the same rules as for characters in double-quotes described in this section except that it shall additionally retain its special meaning as an escape character when followed by '}' and this shall prevent the escaped '}' from being considered when determining the matching '}' (using the rule in 2.6.2 Parameter Expansion).

## Context

**Behavior:** This is the comprehensive rule governing `$` inside double-quotes. It specifies several sub-behaviors:

1. **`$` introduces expansions:** Parameter expansion (`${...}`), command substitution (`$(...)`), and arithmetic expansion (`$((...))`), but NOT dollar-single-quotes (`$'...'`).

2. **`$(...)` command substitution is a nested context:** Characters between `$(` and the matching `)` are NOT affected by the enclosing double-quotes. The tokenizer applies 2.3 Token Recognition recursively to find the matching `)`. This means unquoted spaces, pipes, etc. inside `$(...)` retain their normal meaning even though the `$(...)` is inside double-quotes.

3. **Substring-processing parameter expansions have relaxed quoting:** For the four varieties (`${parameter%word}`, `${parameter%%word}`, `${parameter#word}`, `${parameter##word}`), the double-quotes surrounding the expansion have NO effect on special characters inside `${...}`. Pattern characters in `word` are active.

4. **Other parameter expansions preserve literals:** For non-substring parameter expansions (e.g., `${parameter:-word}`), double-quotes preserve literal values inside `${...}`, except for `"`, `` ` ``, `$`, and `\`. Unescaped double-quote characters inside (other than in embedded command substitutions) produce unspecified behavior.

5. **Backslash inside `${...}`:** Follows the same double-quote rules, plus it can escape `}` to prevent it from being the closing brace.

**Applies to:** All contexts where double-quoted words are expanded.
