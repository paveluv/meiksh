## Extracted Text

> If sh receives a SIGINT signal in insert mode (whether generated by typing the interrupt character or by other means), it shall terminate command line editing with the same effects as described for interrupting command mode; see Command Line Editing (vi-mode).

## Context

This obligation defines the behavior when **SIGINT is received during vi insert mode** in the interactive command line editor.

**Behavior required:** When the shell receives a SIGINT signal while in vi insert mode — whether generated by the user typing the terminal's `interrupt` character (typically Ctrl-C, as set by `stty`) or delivered by other means (e.g., `kill -INT`) — the shell shall terminate command line editing with the **same effects** as the SIGINT behavior defined for command mode.

**Command mode SIGINT effects** (see `utilities/sh.html` tag_20_110_13_03, lines 339–343):
1. Terminate command line editing on the current command line (discard the current input).
2. Reissue the prompt on the next line of the terminal.
3. Reset the command history so that the most recently executed command remains the previous command — meaning the command that was being edited when interrupted is **not** re-entered into the history.

**Conditions:** Only applies when the shell is interactive, vi editing mode is enabled (`set -o vi`), and the shell is in insert mode at the time SIGINT is received.

**Post-interrupt state:** After handling the interrupt, the shell returns to insert mode (since it reissues the prompt and awaits a new command, and `sh` enters insert mode after termination of the previous command).

**Implementation note:** The signal may arrive asynchronously. The handler must safely abort any in-progress editing state, clear the current line buffer, and restore the terminal/prompt to a clean state.
