## Extracted Text

> Parameters can contain arbitrary byte sequences, except for the null byte. The shell shall process their values as characters only when performing operations that are described in this standard in terms of characters.

## Context

The shell's internal parameter storage must support arbitrary byte sequences (excluding the null byte `\0`). This means parameter values are fundamentally byte strings, not character strings.

**Required behavior:**
- Store parameter values as raw byte sequences. The null byte (`\0`) is the only byte that cannot appear in a parameter value (it serves as the C string terminator).
- Character-level interpretation (e.g., multi-byte character boundaries, character class membership, character counting) must only be applied when a specific operation defined by the standard requires character semantics. Such operations include: pattern matching (2.6.6 Pathname Expansion, `case` patterns), `${#parameter}` length counting, `${parameter%pattern}` / `${parameter#pattern}` suffix/prefix removal, and field splitting when IFS contains multi-byte characters.
- In all other contexts (assignment, concatenation, simple expansion `$var` / `${var}`), the shell must preserve the raw bytes without reinterpreting or validating them as characters.

**Conditions:** Applies universally to all parameter types (positional, special, and variables) in all execution contexts.

**Error handling:** The shell must never reject or corrupt a parameter value on the basis that it contains bytes that are not valid characters in the current locale. Invalid byte sequences must be stored and reproduced faithfully.

See 2.5 "Parameters and Variables" in `utilities/V3_chap02.html`.
