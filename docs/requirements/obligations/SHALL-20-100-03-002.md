## Extracted Text

> If the -r option is not specified, <backslash> shall act as an escape character. An unescaped <backslash> shall preserve the literal value of a following <backslash> and shall prevent a following byte (if any) from being used to split fields, with the exception of either <newline> or the logical line delimiter specified with the -d delim option (if it is used and delim is not <newline>); it is unspecified which. If this excepted character follows the <backslash>, the read utility shall interpret this as line continuation. The <backslash> and the excepted character shall be removed before splitting the input into fields. All other unescaped <backslash> characters shall be removed after splitting the input into fields.

## Context

**Behavior required:** When `-r` is NOT specified, backslash acts as an escape character with these rules:

1. **`\\` (backslash-backslash):** The first backslash escapes the second. The pair becomes a single literal backslash.
2. **`\<byte>` where byte is not newline/delimiter:** The backslash preserves the literal value of the following byte AND prevents that byte from being used as an IFS field separator during splitting.
3. **`\<newline>` or `\<delim>`:** This is line continuation. The backslash and the newline/delimiter are both removed, and `read` continues reading the next physical line.
4. **Removal order:**
   - Backslash-newline (continuation) pairs are removed BEFORE field splitting.
   - All other unescaped backslashes are removed AFTER field splitting.

**Unspecified aspect:** When `-d delim` is used and `delim` is not newline, it is unspecified whether `\<newline>` or `\<delim>` triggers line continuation. The implementation must pick one.

**Condition:** When `-r` is not specified (the default).
