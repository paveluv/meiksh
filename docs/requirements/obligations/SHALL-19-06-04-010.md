## Extracted Text

> As an extension, the shell may recognize arithmetic expressions beyond those listed. The shell may use a signed integer type with a rank larger than the rank of signed long. The shell may use a real-floating type instead of signed long as long as it does not affect the results in cases where there is no overflow. If the expression is invalid, or the contents of a shell variable used in the expression are not recognized by the shell, the expansion fails and the shell shall write a diagnostic message to standard error indicating the failure.

## Context

**Behavior:** This obligation covers both permitted extensions and required error handling:

**Permitted extensions (MAY):**
- Recognize additional arithmetic expressions beyond the minimum set
- Use a larger integer type (e.g., `long long` / 64-bit instead of 32-bit `long`)
- Use real-floating types (e.g., `double`) as long as results are identical to integer arithmetic for non-overflow cases

**Required error handling (SHALL):**
- If the arithmetic expression is **invalid** (syntax error, unsupported operator, etc.), the expansion **fails**.
- If a shell variable used in the expression contains a value that is **not recognized** as a valid arithmetic operand (e.g., `x="hello"; echo $((x + 1))`), the expansion **fails**.
- On failure, the shell **must write a diagnostic message to standard error** indicating the failure.
- For non-interactive shells, this failure should cause the shell to exit with a non-zero status (per general expansion failure rules).

**Inputs/Outputs:**
- `$((1 / 0))` → division by zero: write diagnostic to stderr, expansion fails
- `x=abc; $((x + 1))` → non-numeric variable: write diagnostic to stderr, expansion fails
- `$((2 ** 3))` → if `**` is not supported: write diagnostic to stderr, expansion fails
