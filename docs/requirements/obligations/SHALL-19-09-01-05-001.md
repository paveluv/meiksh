## Extracted Text

> If the utility would be executed with file descriptor 0, 1, or 2 closed, implementations may execute the utility with the file descriptor open to an unspecified file. If a standard utility or a conforming application is executed with file descriptor 0 not open for reading or with file descriptor 1 or 2 not open for writing, the environment in which the utility or application is executed shall be deemed non-conforming, and consequently the utility or application might not behave as described in this standard.

## Context

**Behavior:** Two rules about standard file descriptors (0, 1, 2):

1. **MAY open closed descriptors:** If a utility would be executed with fd 0, 1, or 2 closed, the implementation MAY open the descriptor to an unspecified file before executing the utility. This is a defensive measure to prevent utilities from accidentally opening files on these descriptors.

2. **Non-conforming environment:** If fd 0 is not open for reading, or fd 1 or fd 2 is not open for writing, the execution environment is deemed non-conforming and the utility/application might not behave as specified.

**Applies to:** All utility execution (both builtin and external). This is particularly relevant when the user closes descriptors explicitly (e.g., `cmd 0<&- 1>&-`).

**Implementation detail:** meiksh may choose to open `/dev/null` (or similar) on any of fds 0-2 that are closed before exec'ing a child. This is optional but recommended for robustness. The shell should be aware that its own operation requires valid fds 0-2.
