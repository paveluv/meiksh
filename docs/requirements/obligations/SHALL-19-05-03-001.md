## Extracted Text

> Variables shall be initialized from the environment (as defined by XBD 8. Environment Variables and the exec function in the System Interfaces volume of POSIX.1-2024) and can be given new values with variable assignment commands. Shell variables shall be initialized only from environment variables that have valid names. If a variable is initialized from the environment, it shall be marked for export immediately; see the export special built-in. New variables can be defined and initialized with variable assignments, with the read or getopts utilities, with the name parameter in a for loop, with the ${name=word} expansion, or with other mechanisms provided as implementation extensions.

## Context

This defines how shell variables are initialized and the valid sources for creating new variables.

**Required behavior — initialization from environment:**
- At startup, the shell must read environment variables (as inherited via `exec`) and initialize corresponding shell variables from them.
- Only environment variables with valid shell variable names (matching `[A-Za-z_][A-Za-z0-9_]*`) must be imported. Environment entries with invalid names (e.g., containing `=`, `-`, or starting with a digit) must be silently ignored for variable initialization purposes.
- Any variable initialized from the environment must be immediately marked for export (as if `export` had been called on it). This means it will be passed to child processes.

**Valid ways to create new variables:**
1. Variable assignment commands (`VAR=value`)
2. The `read` utility (`read VAR`)
3. The `getopts` utility (`getopts optstring VAR`)
4. The `name` parameter in a `for` loop (`for VAR in ...`)
5. The `${name=word}` expansion form
6. Implementation extensions (shell-specific features)

**Conditions:** Applies during shell initialization and throughout execution.

See 2.5.3 "Shell Variables", the `export` special built-in, XBD 8 "Environment Variables" in `basedefs/V1_chap08.html`, and the `read` utility at `utilities/read.html`.
