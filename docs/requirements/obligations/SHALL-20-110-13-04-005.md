## Extracted Text

> Interpreted as the end of input in sh. This interpretation shall occur only at the beginning of an input line. If end-of-file is entered other than at the beginning of the line, the results are unspecified.

## Context

This obligation defines the behavior of the **terminal `end-of-file` character in vi insert mode** (typically Ctrl-D, as configured by `stty`) for the interactive command line editor.

**Behavior required:**
1. When the `end-of-file` character is entered at the **beginning of an input line** (i.e., the line buffer is empty / cursor is at position 0 with no characters entered), it shall be interpreted as end of input to `sh`. This causes the shell to exit (equivalent to receiving EOF on stdin), subject to the usual shell exit processing (running EXIT traps, etc.).
2. When `end-of-file` is entered at any position **other than** the beginning of the line (i.e., the line buffer contains one or more characters), the results are **unspecified**. The implementation may choose any behavior — common choices include ignoring the character, treating it literally, or alerting the terminal.

**Conditions:** Only applies when the shell is interactive, vi editing mode is enabled (`set -o vi`), and the shell is in insert mode.

**"Beginning of an input line":** This means the cursor is at the start of the command text (after the prompt) and no characters have been entered on this line yet. It does NOT mean the start of a multi-line continuation — it means a fresh, empty command line.

**Relation to `ignoreeof`:** The `set -o ignoreeof` option (see `utilities/sh.html`, tag_20_110_13) may modify the behavior when EOF is received, requiring the user to type `exit` explicitly. If `ignoreeof` is set, the shell should warn rather than exit. This interaction is governed by separate obligations but implementers should be aware of the coupling.
