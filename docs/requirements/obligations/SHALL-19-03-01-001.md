## Context

**Behavior:** After the tokenizer produces a token categorized as type TOKEN (per 2.10.1 Shell Grammar Lexical Conventions), and including any token that itself resulted from a previous alias substitution (recursively), that TOKEN is subject to alias substitution if ALL five conditions listed in sibling obligations (SHALL-19-03-01-002 through 006) are true. This is the master rule establishing the alias substitution trigger point.

**When alias substitution occurs:** It happens AFTER token recognition and token categorization, but BEFORE the grammar processes the token as a command name, reserved word, assignment, etc. It is part of the lexer-to-parser interface.

**Recursive nature:** The alias value itself is re-tokenized, and resulting TOKENs are again checked for alias substitution. This enables alias chaining (e.g., `alias ll='ls -l'` then `alias ls='ls --color'`).

**Applies to:** All shell modes where alias processing is active (typically interactive shells and scripts that explicitly enable aliases).

**Spec reference:** Section 2.3.1, lines 314-315 in `utilities/V3_chap02.html`. Cross-ref: 2.10.1 Shell Grammar Lexical Conventions.
