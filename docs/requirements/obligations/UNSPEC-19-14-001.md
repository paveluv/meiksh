## Extracted Text

> If an attempt is made to use pattern matching notation to match a string that contains one or more bytes that do not form part of a valid character, the behavior is unspecified. Since pathnames can contain such bytes, portable applications need to ensure that the current locale is the C or POSIX locale when performing pattern matching (or expansion) on arbitrary pathnames.

## Context

**Pattern matching with invalid multibyte characters: behavior is unspecified.**

When a string to be matched against a pattern contains bytes that do not form valid characters in the current locale (e.g., invalid UTF-8 sequences), the behavior of pattern matching is **unspecified**. This applies to:
- Pathname expansion (globbing)
- `case` pattern matching
- Parameter expansion patterns (`${var#pattern}`, `${var%pattern}`, etc.)

Since pathnames can contain arbitrary bytes, portable applications should set `LC_ALL=C` or `LC_ALL=POSIX` before performing pattern matching on arbitrary pathnames.

The implementation may choose to: treat invalid bytes as individual single-byte characters, reject them, or produce undefined results. A reasonable approach is to fall back to byte-by-byte matching in the C/POSIX locale when invalid sequences are encountered.
