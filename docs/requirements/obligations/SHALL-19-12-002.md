## Extracted Text

> When a signal for which a trap has been set is received while the shell is waiting for the completion of a utility executing a foreground command, the trap associated with that signal shall not be executed until after the foreground command has completed. When the shell is waiting, by means of the wait utility, for asynchronous commands to complete, the reception of a signal for which a trap has been set shall cause the wait utility to return immediately with an exit status >128, immediately after which the trap associated with that signal shall be taken.

## Context

**Trap execution timing: foreground commands vs. wait built-in.**

Two distinct behaviors are required:

1. **Foreground command running**: When the shell is waiting for a foreground command to complete and a trapped signal arrives, the trap handler **shall not** execute until after the foreground command finishes. The shell must defer trap execution, queuing the signal. Once the foreground command exits, the trap action is then run.

2. **`wait` built-in running**: When the shell is blocked in the `wait` utility waiting for async commands, and a trapped signal arrives, `wait` **shall** return immediately with exit status >128 (typically 128 + signal_number). Immediately after `wait` returns, the trap action **shall** be executed.

Implementation requirements:
- The signal handler must set a flag (not run the trap inline) during foreground waits.
- After `waitpid()` returns for a foreground child, check pending trap flags and execute deferred traps.
- During `wait` built-in execution, arrange for signals to interrupt the blocking wait (e.g., `EINTR` from `waitpid()`), then return >128 and run the trap.
- See `wait` utility (utilities/wait.html) for the exit status specification.
