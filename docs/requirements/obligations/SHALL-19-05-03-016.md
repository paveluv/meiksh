## Extracted Text

> The following variables shall affect the execution of the shell:: Determine the interpretation of sequences of bytes of text data as characters (for example, single-byte as opposed to multi-byte characters), which characters are defined as letters (character class alpha) and <blank> characters (character class blank), and the behavior of character classes within pattern matching. Changing the value of LC_CTYPE after the shell has started shall not affect the lexical processing of shell commands in the current shell execution environment or its subshells. Invoking a shell script or performing exec sh subjects the new shell to the changes in LC_CTYPE .

## Context

This defines the semantics of the `LC_CTYPE` variable.

**Required behavior:**
- `LC_CTYPE` determines how the shell interprets byte sequences as characters. This affects:
  - Single-byte vs. multi-byte character interpretation.
  - Which characters are classified as letters (character class `alpha`) — relevant for variable names, pattern matching `[[:alpha:]]`.
  - Which characters are `<blank>` (character class `blank`) — relevant for tokenization.
  - Behavior of character classes within pattern matching.
- **Immutability constraint:** Changing `LC_CTYPE` after the shell has started must NOT affect the lexical processing (tokenization, parsing) of shell commands in the current shell execution environment or its subshells. The shell's tokenizer/parser must use the `LC_CTYPE` value that was in effect when the shell was initialized.
- A new shell invoked via a shell script or `exec sh` IS subject to the changed `LC_CTYPE` value (it re-initializes).

**Conditions:** The immutability rule means the shell must capture `LC_CTYPE` at startup for its own lexical processing. Child processes and new shell invocations see updated values.

See 2.5.3 "Shell Variables" (`LC_CTYPE`), `exec` built-in, and `utilities/sh.html` in the spec.
