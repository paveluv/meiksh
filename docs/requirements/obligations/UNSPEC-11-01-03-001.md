## Extracted Text

> The controlling terminal is inherited by a child process during a fork() function call. A process relinquishes its controlling terminal when it creates a new session with the setsid() function; other processes remaining in the old session that had this terminal as their controlling terminal continue to have it. Upon the close of the last file descriptor in the system (whether or not it is in the current session) associated with the controlling terminal, it is unspecified whether all processes that had that terminal as their controlling terminal cease to have any controlling terminal. Whether and how a session leader can reacquire a controlling terminal after the controlling terminal has been relinquished in this fashion is unspecified. A process does not relinquish its controlling terminal simply by closing all of its file descriptors associated with the controlling terminal if other processes continue to have it open.

## Context

**Behavior:** Controlling terminal inheritance and relinquishment:
- fork() → child inherits controlling terminal
- setsid() → calling process relinquishes its controlling terminal; old session processes keep it
- Closing all fds to the controlling terminal while other processes still have it → process keeps its controlling terminal
- If the last fd system-wide to the controlling terminal is closed → unspecified whether processes lose their controlling terminal
- Reacquiring a controlling terminal after relinquishment → unspecified

**Conditions:** These rules govern session lifecycle. The shell forks child processes that inherit the controlling terminal. A shell calling setsid() (login shell startup) creates a new session without a controlling terminal and must acquire one.

**Shell relevance:** The shell must understand that forked children inherit the controlling terminal. When the shell creates a new session via setsid(), it must explicitly acquire a controlling terminal.
