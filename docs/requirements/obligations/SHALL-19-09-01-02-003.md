## Extracted Text

> Variable assignments shall be performed as follows:: If the command name is not a special built-in utility or function, the variable assignments shall be exported for the execution environment of the command and shall not affect the current execution environment except as a side-effect of the expansions performed in step 4. In this case it is unspecified: Whether or not the assignments are visible for subsequent expansions in step 4 Whether variable assignments made as side-effects of these expansions are visible for subsequent expansions in step 4, or in the current shell execution environment, or both

## Context

**Behavior:** When the command name is NOT a special built-in utility or function (i.e., it's a regular external command or intrinsic utility), variable assignments MUST be exported for the command's execution environment but MUST NOT affect the current shell environment (except as side-effects of expansion). Example: `VAR=hello /bin/cmd` exports VAR to cmd but VAR is not set in the shell after cmd returns.

Two things are explicitly UNSPECIFIED: (1) whether the assignments are visible to subsequent expansions in step 4 of the same command, and (2) whether side-effect assignments from those expansions persist in the current shell.

**Applies to:** Simple commands with prefix assignments that resolve to external commands or regular builtins (not special builtins, not functions).

**Implementation detail:** Create a temporary environment for the child process with the assigned variables exported. After the command completes, discard them. The typical implementation forks, adds to the child's environ, then execs.

**Spec reference:** Section 2.9.1.2, lines 1764-1769 in `utilities/V3_chap02.html`.
