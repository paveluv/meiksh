## Extracted Text

> The application shall ensure that a double-quote that is not within "$(...)" nor within "${...}" is immediately preceded by a <backslash> in order to be included within double-quotes. The parameter '@' has special meaning inside double-quotes and is described in 2.5.2 Special Parameters.

## Context

**Behavior:** Two rules are combined here:

1. **Application constraint on literal double-quotes:** To include a literal `"` character inside a double-quoted string (outside of `$(...)` and `${...}`), the application must immediately precede it with `\`. The shell is not required to diagnose violations—this is a requirement on the script author, but the shell must correctly handle the `\"` escape (producing a literal `"`).

2. **`"$@"` has special semantics:** Inside double-quotes, the special parameter `@` causes `"$@"` to expand to a separate word for each positional parameter, preserving the exact values (including empty strings and whitespace). If there are no positional parameters, `"$@"` expands to nothing (zero words, not one empty word). This is distinct from `"$*"` which joins positional parameters with the first character of IFS.

**Applies to:** All shell modes during expansion.

**Implementation detail:** The expander must special-case `"$@"` to produce multiple fields even though it appears inside double-quotes. This affects field splitting logic—`"$@"` must bypass the normal "double-quoted = single field" rule.

**Spec reference:** Section 2.2.3, lines 160-162 in `utilities/V3_chap02.html`. See also 2.5.2 Special Parameters for the full `@` semantics.
