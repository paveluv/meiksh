## Extracted Text

> The setting of variables specified by the var operands shall affect the current shell execution environment; see 2.13 Shell Execution Environment. An error in setting any variable (such as if a var has previously been marked readonly) shall be considered an error of read processing, and shall result in a return value greater than one. Variables named before the one generating the error shall be set as described above; it is unspecified whether variables named later shall be set as above, or read simply ceases processing when the error occurs, leaving later named variables unaltered. ## Context

**Behavior required:**
1. **Environment scope:** Variables set by `read` affect the CURRENT shell execution environment (not a subshell). See Section 2.13.
2. **Readonly error handling:** If a var operand names a readonly variable, this is an error. The `read` builtin must return >1 (note: greater than ONE, not just non-zero — this distinguishes it from EOF which returns 1).
3. **Partial assignment on error:** Variables processed BEFORE the error must be set normally. Whether variables AFTER the erroring one are set or left unchanged is unspecified.
4. **Subshell context:** If `read` runs in a subshell or separate execution environment (e.g., in a pipeline like `echo foo | read var`, or in `$(read var)`), the variable assignments do not affect the parent shell's environment.

**Condition:** All invocations of `read`.

**Error examples:**
- `readonly x; read x y` → error on `x`, return >1
- `echo foo | read var` → `var` is set in the subshell but not in the parent
