## Extracted Text

> If job control is disabled (see the description of set -m) when the shell executes an asynchronous AND-OR list, the commands in the list shall inherit from the shell a signal action of ignored (SIG_IGN) for the SIGINT and SIGQUIT signals. In all other cases, commands executed by the shell shall inherit the same signal actions as those inherited by the shell from its parent unless a signal action is modified by the trap special built-in (see trap)

## Context

**Signal inheritance for asynchronous commands and general commands.**

When the shell executes an asynchronous AND-OR list (i.e., a command list terminated with `&`) and job control is disabled (`set +m`, or not an interactive shell with job control):
- The commands in that async list **shall** inherit `SIG_IGN` for both `SIGINT` and `SIGQUIT`. This prevents background jobs from being killed by Ctrl+C or Ctrl+\ intended for the foreground.

In **all other cases** (foreground commands, or async commands when job control is enabled):
- Commands **shall** inherit the same signal dispositions that the shell itself inherited from its parent process, unless a specific signal's action has been modified by the `trap` built-in.

Key implementation points:
- Before `fork()`/`exec()` for an async list with job control disabled, set `SIGINT` and `SIGQUIT` to `SIG_IGN` in the child.
- For all other commands, propagate the shell's inherited signal mask unchanged (except for trap-modified signals, which are reset to default for external commands per section 2.13).
- The `trap` built-in (see 2.29 in utilities/V3_chap02.html) can override signal actions; these overrides apply only within the shell process itself, not to exec'd children (which get default or ignore).
- The `set -m` option (see `set` in utilities/V3_chap02.html) controls whether job control is enabled.
