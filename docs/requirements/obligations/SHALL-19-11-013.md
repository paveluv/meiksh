## Extracted Text

> For each pipeline in a foreground job, if the pipeline is executed while the list is still a foreground job, the set of processes comprising the pipeline, and any processes descended from it, shall all be in the same process group, unless the shell executes some of the commands in the pipeline in the current shell execution environment and others in a subshell environment; in this case the process group ID of the current shell need not change (or cannot change if it is the session leader), and consequently the process group ID that the other processes all share may differ from the process group ID of the current shell (which means that a SIGSTOP, SIGTSTP, SIGTTIN, or SIGTTOU signal sent to one of those process groups does not cause the whole pipeline to stop).

## Context

**Pipeline process group in foreground jobs:** For each pipeline in a foreground job, all processes in the pipeline and their descendants shall be in the same process group. This enables the entire pipeline to be controlled as a unit (e.g., stopped by Ctrl+Z sending SIGTSTP to the foreground process group).

**Exception for mixed execution environments:** If the shell executes some pipeline commands in the current shell execution environment (e.g., builtins) and others in subshell environments, the shell's own process group ID need not change (especially if the shell is the session leader, where `setpgid` would fail). In this case, the forked processes share a process group that may differ from the shell's, meaning a stop signal to the pipeline's process group may not stop the builtin running in the shell process. This is an acknowledged limitation.

The implementation should use `setpgid()` to place all forked pipeline children into a common process group. For builtins executed in the current environment, the shell process cannot change its own process group if it's the session leader.
