## Extracted Text

> All of the requirements and effects of quoting on ordinary, shell special, and special pattern characters shall apply to escaping in this context, except where specified otherwise. (Situations where this applies include word expansions when a pattern used in pathname expansion is not present in the original word but results from an earlier expansion, or the argument to the find -name or -path primary as passed to find, or the pattern argument to the fnmatch() and glob() functions when FNM_NOESCAPE or GLOB_NOESCAPE is not set in flags, respectively.)

## Context

**Quoting/escaping rules apply uniformly to pattern escaping in non-shell-quoting contexts.**

All quoting requirements and effects on ordinary, shell special, and special pattern characters **shall** apply to `\`-escaping in non-shell-quoting pattern contexts, except where otherwise specified. This means the same characters that need quoting in shell source also need escaping with `\` in these contexts to be treated literally.

This applies in situations such as:
- Pathname expansion where the pattern was produced by an earlier expansion (not in the original word)
- The `find -name` or `-path` argument
- Arguments to `fnmatch()` and `glob()` without `FNM_NOESCAPE`/`GLOB_NOESCAPE`

For the shell specifically, this means if a variable expands to contain `*`, `?`, or `[`, those characters are treated as pattern characters during pathname expansion (unless the expansion was double-quoted). The refs to `find`, `fnmatch()`, and `glob()` are informative context — the shell need not implement those.
