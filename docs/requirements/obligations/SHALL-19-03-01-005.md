## Extracted Text

> After a token has been categorized as type TOKEN (see 2.10.1 Shell Grammar Lexical Conventions), including (recursively) any token resulting from an alias substitution, the TOKEN shall be subject to alias substitution if all of the following conditions are true:: The TOKEN did not either fully or, optionally, partially result from an alias substitution of the same alias name at any earlier recursion level.

## Context

**Behavior:** Alias substitution condition 4 (recursion guard): The TOKEN must not have resulted (fully or, optionally, partially) from an alias substitution of the SAME alias name at any earlier recursion level. This prevents infinite recursion.

**Applies to:** All shell modes during alias processing.

**"Fully or, optionally, partially":** The spec requires that a token that fully came from substituting alias `X` must not trigger alias `X` again. Implementations MAY also block partial matches (e.g., if alias `X` expanded to text that was combined with adjacent characters to form a new token that equals `X`).

**Implementation detail:** Maintain a set/stack of "active alias names" during recursive alias expansion. Before expanding an alias, check if that name is already in the active set. If so, skip expansion. Pop the name when returning from that recursion level.

**Example:** `alias ls='ls --color'` — when `ls` expands to `ls --color`, the re-tokenized `ls` is checked for alias expansion, but since `ls` is in the active-alias set, it is NOT re-expanded (preventing infinite recursion).

**Spec reference:** Section 2.3.1, lines 327-328 in `utilities/V3_chap02.html`.
