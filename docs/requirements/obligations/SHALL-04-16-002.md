## Extracted Text

> Each filename in the pathname is located in the directory specified by its predecessor (for example, in the pathname fragment a/b, file b is located in the directory specified by a). Pathname resolution shall fail if this cannot be accomplished. If the pathname begins with a <slash>, the predecessor of the first filename in the pathname shall be taken to be the root directory of the process (such pathnames are referred to as "absolute pathnames"). If the pathname does not begin with a <slash>, the predecessor of the first filename of the pathname shall be taken to be either the current working directory of the process or for certain interfaces the directory identified by a file descriptor passed to the interface (such pathnames are referred to as "relative pathnames").

## Context

This defines the fundamental distinction between absolute and relative pathnames, which the shell must handle correctly everywhere:

- **Absolute pathnames** (starting with `/`): resolution starts at the root directory. The shell encounters these in command paths, redirection targets, `cd` arguments, `PATH` entries, etc.
- **Relative pathnames** (not starting with `/`): resolution starts at the current working directory (CWD). This is critical for the shell because CWD is tracked via the `PWD` variable and `cd` builtin.

The shell must ensure that when it performs operations like `cd foo/bar`, each component (`foo`, then `bar`) is resolved in the directory found by its predecessor. If any component cannot be found, the operation must fail. The shell relies on the OS kernel for actual pathname resolution via syscalls like `open()`, `stat()`, `chdir()`, etc.
