## Extracted Text

> Otherwise, the shell shall terminate in the same manner as for an exit command with no operands, unless the last command the shell invoked was executed without forking, in which case the wait status seen by the parent process of the shell shall be the wait status of the last command the shell invoked. See the exit utility in 2.15 Special Built-In Utilities.

## Context

**Behavior:** For all exit conditions not covered by the specific exit codes (0, 1-125, 126, 127, 128), the shell shall terminate as follows:

1. **Default termination:** The shell shall terminate as if `exit` were called with no operands. Per the `exit` built-in specification, this means the exit status is the value of `$?` (the exit status of the last command executed).

2. **Optimization exception (exec without fork):** If the last command the shell invoked was executed without forking (i.e., the shell optimized by exec'ing the last command directly rather than fork+exec), the wait status seen by the parent shall be the wait status of that last command. This allows the shell to optimize tail calls without losing the exit status information.

**Applies to:** Normal shell termination (reaching end of input, `exit` command, etc.) in both interactive and non-interactive modes.

**Details:** This is the "catch-all" termination rule. It covers:
- Normal end of script (EOF reached).
- Explicit `exit` command without arguments.
- Interactive shell receiving EOF (e.g., Ctrl-D).
