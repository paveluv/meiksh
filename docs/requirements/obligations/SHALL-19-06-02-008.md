## Extracted Text

> Indicate Error if Null or Unset. If parameter is unset or null, the expansion of word (or a message indicating it is unset if word is omitted) shall be written to standard error and the shell exits with a non-zero exit status. Otherwise, the value of parameter shall be substituted. An interactive shell need not exit.

## Context

**Behavior:** The `${parameter:?word}` expansion (Indicate Error if Null or Unset):
- If `parameter` is **unset or null**:
  1. Expand `word` and write the result to **standard error**. If `word` is omitted, write a default message indicating the parameter is unset (e.g., `parameter: parameter not set`).
  2. The shell must **exit with a non-zero exit status** (for non-interactive shells).
- If `parameter` is **set and non-null**: substitute the value of `parameter`; no error.

**Interactive shell exception:** An interactive shell **need not exit** when this error is triggered. It should still write the error message to stderr, but may continue accepting commands.

**No-colon variant:** `${parameter?word}` only tests for unset (not null) — see SHALL-19-06-02-010.

**Inputs/Outputs:**
- `${REQUIRED_VAR:?must be set}` → if unset/null, writes `REQUIRED_VAR: must be set` to stderr and exits
- `${REQUIRED_VAR:?}` → if unset/null, writes a default "parameter not set" message to stderr and exits

**Error handling:** This IS the error handling mechanism — it's designed to abort scripts when required variables are missing.

**Spec reference:** See 2.6.2 Parameter Expansion in utilities/V3_chap02.html, lines 815–818.
