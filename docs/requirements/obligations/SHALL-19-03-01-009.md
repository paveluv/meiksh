## Extracted Text

> When used as specified by this volume of POSIX.1-2024, alias definitions shall not be inherited by separate invocations of the shell or by the utility execution environments invoked by the shell; see 2.13 Shell Execution Environment .

## Context

**Behavior:** Alias definitions are NOT inherited by:
1. **Separate shell invocations** — running `sh` or `bash` as a child process does not pass aliases to it.
2. **Utility execution environments** — commands executed by the shell (via `exec`, `fork+exec`, etc.) do not receive the shell's alias table.

Aliases are local to the shell process that defines them. They are part of the shell's internal state, not the environment.

**Applies to:** All shell modes.

**Implementation detail:** The alias table is stored in shell-internal memory only. It is NOT exported via environment variables or any other IPC mechanism. When the shell forks a subshell, the subshell inherits a copy of the alias table (since it's a fork), but a separately invoked shell (via `sh script.sh` or similar) starts with a clean alias table (plus any predefined aliases per SHALL-19-03-01-008).

**Spec reference:** Section 2.3.1, lines 358-359 in `utilities/V3_chap02.html`. Cross-ref: 2.13 Shell Execution Environment.
