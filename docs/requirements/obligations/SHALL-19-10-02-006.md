## Extracted Text

> [Third word of for and case] [case only] When the TOKEN is exactly the reserved word in, the token identifier for in shall result. Otherwise, the token WORD shall be returned. [for only] When the TOKEN is exactly the reserved word in or do, the token identifier for in or do shall result, respectively. Otherwise, the token WORD shall be returned. (For a. and b.: As indicated in the grammar, a linebreak precedes the tokens in and do. If <newline> characters are present at the indicated location, it is the token after them that is treated in this fashion.)

## Context

**Grammar Rule 6 (Third word of `for` and `case`):** This rule handles the token expected after the NAME/word in `for` and `case` constructs:

- **`case` context:** After `case WORD`, the next non-linebreak token must be `in`. If the TOKEN is exactly `in`, produce the `In` token identifier. Otherwise, produce WORD (syntax error).
- **`for` context:** After `for NAME`, the next non-linebreak token may be `in` or `do`. If the TOKEN is exactly `in`, produce `In`. If exactly `do`, produce `Do`. Otherwise, produce WORD.

Key detail: The grammar allows linebreaks (`\n` sequences) between the NAME/WORD and this token. The rule applies to the first non-linebreak token after those linebreaks. For example, `for x \n in ...` is valid—the `in` after the newline is still recognized via this rule.

The implementation must handle the linebreak lookahead correctly to apply this rule to the right token.
