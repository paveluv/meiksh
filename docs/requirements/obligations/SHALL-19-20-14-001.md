## Extracted Text

> If there are no arguments, or only null arguments, eval shall return a zero exit status; otherwise, it shall return the exit status of the command defined by the string of concatenated arguments separated by <space> characters, or a non-zero exit status if the concatenation could not be parsed as a command and the shell is interactive (and therefore did not abort).

## Context

**Behavior:** Exit status rules for the `eval` special built-in:

1. **No arguments or only null arguments:** `eval` (with no args) or `eval "" ""` shall return exit status 0.
2. **Valid command string:** When arguments are provided, they are concatenated with spaces, parsed, and executed. The exit status shall be the exit status of the resulting command.
3. **Parse error (interactive shell):** If the concatenated string cannot be parsed as a valid command and the shell is interactive (which does not abort), a non-zero exit status shall be returned.
4. **Parse error (non-interactive shell):** A non-interactive shell would abort on parse error per 2.8.1 Consequences of Shell Errors, so the non-zero return case does not arise — the shell exits instead.

**Applies to:** All shell modes. The parse-error handling differs between interactive (returns non-zero) and non-interactive (aborts).

**Implementation detail:** The arguments to eval are joined with single space separators (e.g., `eval "a=1" "b=2"` evaluates `a=1 b=2`). The resulting string is then tokenized, parsed, and executed as if it were shell input.
