## Extracted Text

> The following option shall be supported:: Write to standard output a list of commands associated with each condition operand. The behavior when there are no operands is specified in the DESCRIPTION section. The shell shall format the output, including the proper use of quoting, so that it is suitable for reinput to the shell as commands that achieve the same trapping results for the specified set of conditions. If a condition operand is a condition corresponding to the SIGKILL or SIGSTOP signal, and trap -p without any operands would not include it in the set of conditions for which it writes output, the behavior is undefined if the output is reinput to the shell.

## Context

The `trap -p [condition...]` option must write to standard output the commands associated with each specified condition (signal). The detailed behavior is:

1. **With condition operands** (`trap -p SIGINT SIGTERM`): write the trap commands for each specified condition.
2. **Without operands** (`trap -p`): the behavior is specified in the DESCRIPTION section of trap — it writes all currently set traps.

**Output format requirements**:
- The output must be formatted with proper quoting so that it can be **re-input to the shell** to achieve the same trapping results. For example, if `trap 'echo caught' INT` was set, `trap -p INT` might output `trap -- 'echo caught' INT`.
- The shell must handle quoting correctly so that special characters in trap commands survive re-parsing.

**Undefined behavior for SIGKILL/SIGSTOP**: If a condition operand corresponds to SIGKILL or SIGSTOP, and `trap -p` (without operands) would not include it in its output, then the behavior is **undefined** if the output of `trap -p SIGKILL` is re-input to the shell. (SIGKILL and SIGSTOP cannot be trapped, so they normally don't appear in trap output.)

See 2.29 trap in utilities/V3_chap02.html#tag_19_29 (both OPTIONS and DESCRIPTION sections).
