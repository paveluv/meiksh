## Extracted Text

> When a foreground job is created, or a background job is brought into the foreground by the fg utility, if the shell has a controlling terminal it shall set the foreground process group ID associated with the terminal as follows:: If the job was originally created as a foreground job, the foreground process group ID shall be set as follows when each pipeline in the job is executed: If the shell is not itself executing, in the current shell execution environment, all of the commands in the pipeline, the foreground process group ID shall be set to the process group ID that is shared by the other processes executing the pipeline (see above). If all of the commands in the pipeline are being executed by the shell itself in the current shell execution environment, the foreground process group ID shall be set to the process group ID of the shell.

## Context

**Foreground PGID for originally-foreground jobs (parent obligation):** When a foreground job was originally created as a foreground job, the foreground process group ID is set per-pipeline as each pipeline in the job executes:

1. **Shell not executing all commands in current environment:** If the shell forks child processes for any command in the pipeline, set the foreground process group ID to the process group ID shared by those child processes (the pipeline's process group).
2. **Shell executing all commands itself:** If all commands in the pipeline are builtins running in the current shell environment (no forks), set the foreground process group ID to the shell's own process group ID.

This handles the distinction between pipelines with external commands (which have their own process group) and pipelines of pure builtins (which run in the shell's process group). See SHALL-19-11-019 for the detailed version.
