## Extracted Text

> A single-shift encoding (where each character not in the initial shift state is preceded by a shift code) can be defined in the charmap format if each shift-code/character sequence is considered a multi-byte character, defined using the concatenated-constant format described in 6.4 Character Set Description File. If the implementation supports a character encoding of this type, all of the standard utilities shall support it. A locking-shift encoding (where the state of the character is determined by a shift code that may affect more than the single character following it) could be defined with an extension to the charmap format described in 6.4 Character Set Description File.

## Context

**Behavior:** If the platform supports single-shift character encodings (each shifted character preceded by a shift code), then ALL standard utilities — including the shell — must support that encoding. Locking-shift encodings may be supported via charmap extensions.

**Shell relevance:** If a single-shift encoding is the active locale's encoding, the shell must handle it correctly. Each shift-code + character byte sequence is treated as a multi-byte character. The shell would encounter this through the C library's multi-byte functions (`mbtowc`, `mbrtowc`, etc.). In practice, this primarily affects East Asian encodings like ISO-2022 variants, which are rare in modern shell usage.
