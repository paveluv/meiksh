## Extracted Text

> Within the backquoted style of command substitution, if the command substitution is not within double-quotes, <backslash> shall retain its literal meaning, except when followed by: '$', '`', or <backslash>. See 2.2.3 Double-Quotes for the handling of <backslash> when the command substitution is within double-quotes. The search for the matching backquote shall be satisfied by the first unquoted non-escaped backquote; during this search, if a non-escaped backquote is encountered within a shell comment, a here-document, an embedded command substitution of the $(commands) form, or a quoted string, undefined results occur. A quoted string that begins, but does not end, within the "`...`" sequence produces undefined results.

## Context

**Behavior:** Backquote-style command substitution (`` `commands` ``) has special backslash handling:

1. **Outside double-quotes:** Within `` `...` ``, backslash (`\`) retains its literal meaning EXCEPT when followed by `$`, `` ` ``, or `\`. In those three cases, the backslash is consumed as an escape character (e.g., `\$` → literal `$`, `` \` `` → literal `` ` ``, `\\` → literal `\`).
2. **Inside double-quotes:** When the backquoted substitution is itself within `"..."`, the backslash rules from 2.2.3 Double-Quotes apply (backslash is special before `$`, `` ` ``, `"`, `\`, and newline).
3. **Matching backquote search:** The closing `` ` `` is the first unquoted, non-escaped backquote encountered. The parser must not look inside comments, here-documents, `$(...)` substitutions, or quoted strings for the matching backquote — encountering an unescaped backquote in those contexts produces undefined results.
4. **Unterminated quoted strings:** A quoted string that starts but doesn't end within the `` `...` `` sequence produces undefined results.

**Spec reference:** See 2.6.3 Command Substitution in utilities/V3_chap02.html, lines 1113–1119. See 2.2.3 Double-Quotes (#tag_19_02_03) for double-quote backslash rules.
