## Extracted Text

> Matching shall be based on the bit pattern used for encoding the character, not on the graphic representation of the character. This means that if a character set contains two or more encodings for a graphic symbol, or if the strings searched contain text encoded in more than one codeset, no attempt is made to search for any other representation of the encoded symbol. If that is required, the user can specify equivalence classes containing all variations of the desired graphic symbol.

## Context

**Behavior:** Matching shall be based on the bit pattern (encoding) used for the character, not on its graphic representation. If a character set has multiple encodings for the same glyph, or if text contains mixed codesets, no attempt is made to search for alternate representations. Users can use equivalence classes to match variant encodings of the same graphic symbol.

**Shell relevance:** The shell's pattern matching (glob, `case`, bracket expressions) must compare bytes/codepoints, not visual appearances. For example, if a Unicode string contains a precomposed `é` (U+00E9) and a decomposed `é` (U+0065 U+0301), a bracket expression `[é]` matches only the form that has the same byte encoding as the literal `é` in the pattern. The shell must not perform Unicode normalization during matching.
