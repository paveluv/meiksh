## Extracted Text

> When a process writes one or more bytes to a terminal device file, they are processed according to the c_oflag field (see 11.2.3 Output Modes). The implementation may provide a buffering mechanism; as such, when a call to write() completes, all of the bytes written have been scheduled for transmission to the device, but the transmission has not necessarily completed. See write() for the effects of O_NONBLOCK on write().

## Context

**Behavior:** When bytes are written to a terminal device, they are processed according to the output modes (`c_oflag`). The implementation **may** buffer output, so a successful `write()` return means the data has been scheduled for transmission but not necessarily physically sent to the terminal yet.

**Conditions:** Applies to all `write()` calls on terminal device files. Output processing (e.g., OPOST, ONLCR) is applied based on `c_oflag`. The O_NONBLOCK flag on the file descriptor affects `write()` behavior as described in the `write()` specification.

**Shell relevance:** The shell writes output to the terminal (prompts, command output). Because output may be buffered, the shell cannot assume that data is immediately visible after `write()` returns. This matters for prompt display timing and when the shell needs to ensure output is flushed before changing terminal modes (use `tcdrain()` or TCSADRAIN with `tcsetattr()`).

**Spec reference:** POSIX.1-2024, §11.1.8 Writing Data and Output Processing (tag_11_01_08), `basedefs/V1_chap11.html` lines 210–214. See also §11.2.3 Output Modes.
