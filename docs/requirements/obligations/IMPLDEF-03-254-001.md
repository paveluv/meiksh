## Extracted Text

> A string that is used to identify a file. In the context of POSIX.1-2024, a pathname may be limited to {PATH_MAX} bytes, including the terminating null byte. It has optional beginning <slash> characters, followed by zero or more filenames separated by <slash> characters. A pathname can optionally contain one or more trailing <slash> characters. Multiple successive <slash> characters are considered to be the same as one <slash>, except it is implementation-defined whether the case of exactly two leading <slash> characters is treated specially.

## Context

**Definition — direct shell relevance.** Defines "pathname" as a string identifying a file: optional leading slashes, followed by zero or more filenames separated by slashes. Key rules:
- May be limited to `{PATH_MAX}` bytes including NUL terminator
- Multiple consecutive slashes are equivalent to one slash
- **Exception:** Exactly two leading slashes (`//foo`) is implementation-defined — it may be treated specially (e.g., network paths on some systems)
- Trailing slashes are allowed

**Shell impact:** Pathnames are pervasive in shell operations:
- `$PATH` contains colon-separated pathnames for command lookup
- `cd`, `pwd`, and all file-related builtins use pathnames
- Filename expansion (globbing) generates pathnames
- Tilde expansion (`~user`) produces pathnames
- The shell must handle the `//` special case when canonicalizing paths (e.g., in `cd` with `-P`)
- `{PATH_MAX}` limits affect command lookup and `cd` operations

**Design decision:** Whether `meiksh` treats exactly two leading slashes specially. Most Unix systems treat `//` the same as `/`, but some (e.g., Cygwin, some network filesystems) do not.
