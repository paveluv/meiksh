## Extracted Text

> If the job was originally created as a foreground job, the foreground process group ID shall be set as follows when each pipeline in the job is executed: If the shell is not itself executing, in the current shell execution environment, all of the commands in the pipeline, the foreground process group ID shall be set to the process group ID that is shared by the other processes executing the pipeline (see above). If all of the commands in the pipeline are being executed by the shell itself in the current shell execution environment, the foreground process group ID shall be set to the process group ID of the shell.

## Context

**Foreground PGID for originally-foreground jobs (detail):** Per-pipeline rules for setting the foreground process group:

1. **Mixed execution (some forked, some builtin):** If the shell is not executing ALL commands in the pipeline in the current shell environment (i.e., at least one command is forked), set the foreground PGID to the process group ID shared by the forked processes (`tcsetpgrp(fd, pipeline_pgid)`).
2. **All-builtin execution:** If all commands are being executed by the shell itself in the current shell environment, set the foreground PGID to the shell's own process group ID (`tcsetpgrp(fd, shell_pgid)`).

This ensures the terminal is always associated with whichever process group is actually running commands. The implementation must determine per-pipeline whether any forks occurred to choose the correct PGID.
